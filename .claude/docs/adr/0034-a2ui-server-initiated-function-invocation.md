## Context

ADR-0031 mapped every wire error to `VALIDATION_FAILED` and **reserved** the `INVALID_FUNCTION_CALL` arm because the **repo** had no server-initiated function-call path (our `FUNCTION` emits are render-time *binding-evaluation* failures → `VALIDATION_FAILED`, correctly). That was **repo-absence**, not **spec-absence**: **A2UI v1.0 DOES define the surface** — the earlier "no surface" reading conflated the two. The user authorized building it. These shapes are **A2UI v1.0 spec facts** (host-grounded verbatim from a2ui.org/specification/v1.0-a2ui/ by team-lead — Constraint **C1**: attributed to the spec, not our invention):

1. **Inbound — a NEW `callFunction` envelope** (server→client RPC). v1.0: *"This message is sent by the server to execute a function registered on the client."* `{ version, functionCallId (TOP-LEVEL, required), wantResponse? (default false), callFunction: { call, args? } }` — e.g. `{version:'v1.0', functionCallId:'get_device_resolution_123', wantResponse:true, callFunction:{call:'getScreenResolution', args:{screenIndex:0}}}`.
2. **`functionCallId` is TOP-LEVEL** (sibling to `callFunction`). v1.0: *"The client MUST copy this ID verbatim into the subsequent `functionResponse` or error message."*
3. **Outbound success — a NEW `functionResponse`** (client→server): `{ version, functionResponse: { functionCallId (copied), call, value } }`.
4. **Outbound reject — `INVALID_FUNCTION_CALL`** (the arm ADR-0031 modeled): `{ version, error: { code:'INVALID_FUNCTION_CALL', message, functionCallId (copied) } }` — `functionCallId` required, `surfaceId` excluded. **Trigger (verbatim):** *"If the requested function is configured in the catalog as `clientOnly`, or if the function is not registered at all, the client MUST immediately reject the call and return a client-to-server error message with code: `INVALID_FUNCTION_CALL`."*
5. **`callableFrom` is a v1.0 string ENUM** (`clientOnly`/`remoteOnly`/`clientOrRemote`) — **not** the boolean ADR-0031 assumed. v1.0: *"If omitted, it defaults to `clientOnly`."* So an undeclared function is **spec-defined** as not server-invocable (the safe migration is the spec's own default, not our choice).
6. **The two surfaces are SPEC-DISTINCT.** v1.0: UI bindings (`{call}` in DynamicString/`checks`) *"are invoked locally during component rendering — no network round-trip"*; `callFunction` is *"an envelope-level message requiring `functionCallId` tracking, bidirectional RPC."* **Same registered functions, different invocation surface** — reuse the registry, do **not** conflate. (So ADR-0031's binding-eval → `VALIDATION_FAILED` mapping **STANDS** — this ADR does not reopen it; it activates the *separate* reserved arm.)

The seams this builds on: `dispatch.ts` (a pure switch over 5 inbound envelope keys — `callFunction` is the 6th); `DispatchHandlers` (the host-injected handler interface); `renderer.ts #emit` (the single client→server chokepoint, LLD-C13); `catalog/functions.ts` `catalogFunctions` (the shared pure-impl registry, ADR-0026) + `catalog.ts` `FunctionDef`; the `CatalogRegistry` (the renderer's registered catalogs).

## Decision

We add the server-initiated `callFunction` RPC in a **new `renderer/call-function.ts` (LLD-C14)**, reusing ADR-0026's function registry under a `callableFrom` gate, emitting `functionResponse` (success) or activating the reserved `INVALID_FUNCTION_CALL` arm (reject). The **binding-eval surface (`functions.ts`/ADR-0026) is untouched** — the two surfaces stay distinct (fact 6).

1. **Protocol shapes.** `protocol.ts` gains: `A2uiCallFunction { call: string; args?: Record<string, unknown> }`; the inbound envelope `{ version; functionCallId: string; wantResponse?: boolean; callFunction: A2uiCallFunction }` added to `A2uiServerMessage`; the outbound `A2uiFunctionResponse { functionCallId: string; call: string; value: unknown }` + its `{ version; functionResponse }` envelope added to `A2uiClientMessage`. The `INVALID_FUNCTION_CALL` `A2uiWireError` arm is **unchanged** (ADR-0031 already models it — now reachable).
2. **Catalog `callableFrom` (corrects ADR-0031's boolean).** `FunctionDef` gains `callableFrom: 'clientOnly' | 'remoteOnly' | 'clientOrRemote'`. The loader **defaults an undeclared `callableFrom` to `'clientOnly'`** — this is the **v1.0 spec default** (fact 5: *"If omitted, it defaults to `clientOnly`"*), not our choice; it happens to be least-authority (a function is not server-invocable unless explicitly opted in), which is why the migration is safe. The default catalog's `required`/`email`/`regex` are local validators → **`clientOnly`** (declared explicitly in `catalog.json` for clarity, though the spec default would suffice). `returns` is retained (the v1.0 `returnType`; the loader keeps the field).
3. **Dispatch — the 6th envelope route.** `dispatch.ts` adds a `callFunction` case (version-gated like the other five, SPEC-R13) and `DispatchHandlers` gains `callFunction(body, version)`. `callFunction` is **envelope-level** (no `surfaceId`) — the handler is host-level, not surface-scoped.
4. **The invoke mechanism (`call-function.ts`, LLD-C14) — reuse the registry, thin gated dispatch.** On a `callFunction`:
   a. **Look up `callFunction.call`** across the renderer's **registered catalogs** (`CatalogRegistry`) — the active-catalog lookup. *(Single-catalog default = unambiguous; a multi-catalog name collision resolves first-registered-match — flagged as a minor edge, below.)*
   b. **`callableFrom` gate:** reject (`INVALID_FUNCTION_CALL`) if the function is **unregistered** in every catalog **OR** its `callableFrom === 'clientOnly'`. Accept (`remoteOnly`/`clientOrRemote`).
   c. **Invoke the shared pure impl** `catalogFunctions[call](args)` (from `catalog/functions.ts`, ADR-0026) — **NOT** `functions.ts evaluate` (no `@`-system dispatch, no recursive `resolveValue`; this is a flat RPC). The two surfaces share the *registry*, not the dispatch.
   d. **Args are CONCRETE LITERALS** (fork 1) — passed straight to the impl. `callFunction` carries **no `surfaceId`/data model**, so there is nothing to resolve a `{path}`/`{call}` arg against; the spec's "args per the catalog schema" means concrete values the server provides. *(This is the key non-conflation with the binding-eval surface, which DOES resolve args reactively.)*
5. **Response / reject emission (via `#emit`, LLD-C13).**
   - **Success + `wantResponse === true`** → emit `{ version, functionResponse: { functionCallId, call, value } }` (`functionCallId` copied verbatim).
   - **Success + `wantResponse` false/absent** → **fire-and-forget** (invoke, emit nothing).
   - **Reject** (clientOnly/unregistered) → emit `INVALID_FUNCTION_CALL` + `functionCallId` **always** (it is an *error*, not a value-response; not gated on `wantResponse` — the server must learn its invocation was invalid). `surfaceId` is **excluded** (the contextID tie; `toWireError`'s `INVALID_FUNCTION_CALL` arm).
6. **The binding-eval surface is UNCHANGED.** `functions.ts evaluate` (DynamicString/`checks`, ADR-0026) does **not** read `callableFrom` — a local binding is the client surface, and `clientOnly`/`clientOrRemote` functions remain usable in bindings (so `required`/`email`/`regex` in `checks` keep working). `callableFrom` is read **only** by the server-invoke path (clause 4). `@index` is a **system** binding-helper (`@`-prefixed, not in `catalog.functions`) → a server `callFunction:'@index'` is **unregistered → rejected** (correct: a collection-scope helper is never server-invocable).

### Forks ruled

- **Fork 1 — args = CONCRETE LITERALS** (not `resolveValue`-resolved). `callFunction` is surfaceless (no data model); there is nothing to resolve `{path}` against, and the spec scopes args to "the fn's catalog schema" (concrete typed values). Resolving would invent a surface binding the RPC does not have, and would conflate the two surfaces (fact 6). *(If a future surface-scoped server-invoke is added, that is a separate envelope with a `surfaceId`.)*
- **Fork 2 — SHARE the registry, separate thin dispatch.** The server path reuses `catalogFunctions` (impls) + `catalog.functions` (metadata/`callableFrom`) — one source of truth for "what functions exist." It does **not** reuse `evaluate` (the binding dispatcher) — it has its own flat gated call (no `@`-system, no recursive args). A parallel registry is rejected (drift; two sources for the same functions).
- **Fork 3 — `callableFrom` default = `clientOnly`; existing fns → `clientOnly`; `@index` rejects naturally.** Least-authority default: a function is server-invocable only if explicitly `remoteOnly`/`clientOrRemote`. So `required`/`email`/`regex` (validators) → `clientOnly` (bindings keep working; server `callFunction` rejects them — correct). `@index` (system, not a catalog function) → unregistered for `callFunction` → rejects. **Migration is non-breaking** (the binding-eval path ignores `callableFrom`; only the new server path reads it).
- **Fork 4 — `wantResponse` honored.** `true` → `functionResponse(value)`; `false`/absent → fire-and-forget. The **reject** is emitted **regardless** of `wantResponse` (an error is not a value-response).
- **Fork 5 — a throwing invoke → `INVALID_FUNCTION_CALL` + `functionCallId`, NON-FATAL** (spec gap — ruled). The spec names two triggers (clientOnly/unregistered); a *throwing* server-invocable impl is an undocumented runtime fault. With exactly two wire error codes and `functionResponse` carrying no error field, `INVALID_FUNCTION_CALL` (the function-error code, carrying the `functionCallId` the server correlates on) is the nearest conformant fit; the throw is **caught** (never tears down the renderer, SPEC-N4). This mirrors ADR-0026's binding-eval "throwing impl → `FUNCTION`, non-fatal." Recorded as a spec-silence design decision.

## Acceptance criteria (round-trip; the load-bearing gate)

- **AC1 — happy path.** A registered `clientOrRemote` (or `remoteOnly`) function invoked by `callFunction{ functionCallId:'fc1', wantResponse:true, callFunction:{call:'getScreenResolution', args:{…}} }` → the renderer emits `{ functionResponse: { functionCallId:'fc1', call:'getScreenResolution', value:<result> } }`. **The fixture MUST use a function name that does NOT collide with a default-catalog `clientOnly` function** (a fresh `getScreenResolution`, not a shadowed `required`) — the happy path must NOT depend on overriding/shadowing a `clientOnly` declaration (Amendment §collision; the prior build drove this AC by shadowing `required`, a test forcing a security semantic).
- **AC2 — reject `clientOnly`.** `callFunction` for a `clientOnly` function (e.g. `required`) → `{ error: { code:'INVALID_FUNCTION_CALL', functionCallId:'fc1', message } }`; **`surfaceId` ABSENT** (contextID tie); the function is **not** invoked.
- **AC3 — reject unregistered.** `callFunction` for an unknown name (and for `@index`) → `INVALID_FUNCTION_CALL` + `functionCallId`.
- **AC4 — `wantResponse` honored.** `wantResponse:false` (or absent) on a successful invoke → **no** `functionResponse` emitted; `functionCallId` is copied verbatim in every emitted message (AC1/AC2/AC3).
- **AC5 — the two surfaces stay distinct.** A `checks`/DynamicString binding using `required`/`email`/`regex` (now `clientOnly`) **still evaluates locally** (no regression of ADR-0026/0029) — proving `callableFrom` gates only the server path.
- **AC6 — throwing invoke (fork 5).** A registered server-invocable function whose impl throws → `INVALID_FUNCTION_CALL` + `functionCallId`, renderer intact (SPEC-N4).
- **AC7 — `clientOnly` is a HARD FLOOR (the collision semantic; Amendment).** Given a function name declared `clientOnly` in one registered (active) catalog **and** `clientOrRemote`/`remoteOnly` in a sibling registered catalog, a server `callFunction` for it is **STILL rejected** (`INVALID_FUNCTION_CALL` + `functionCallId`) — most-restrictive-wins; the permissive sibling does **not** loosen the `clientOnly` guard, and the result is **independent of registration order**. (Negative control: assert the function is **not** invoked.)

## Consequences

- **`INVALID_FUNCTION_CALL` is now reachable** — ADR-0031's reserved arm activates with a real, conformant trigger (a server `callFunction` the client rejects), carrying the `functionCallId` the tie requires. ADR-0031's `VALIDATION_FAILED`-for-binding-eval mapping is **unchanged** (the two surfaces are distinct).
- **`callableFrom` is a new catalog field with a safe default** — `clientOnly` by default means existing catalogs are non-breaking AND a server cannot invoke a function unless it is explicitly opted in (`remoteOnly`/`clientOrRemote`). This corrects ADR-0031's boolean assumption to the v1.0 enum.
- **The two function surfaces are explicitly separate** — binding-eval (reactive, surface-scoped, `evaluate`) vs server-invoke (flat, surfaceless, concrete args, gated). They share the *registry* only. A future maintainer must not route one through the other.
- **The default catalog ships no server-invocable function yet** — `required`/`email`/`regex` are all `clientOnly`, so every `callFunction` against the default catalog **rejects** until a project catalog registers a `remoteOnly`/`clientOrRemote` function. The AC1 happy-path test registers such a function (a project-catalog fixture); whether the *default* catalog should ship one (e.g. a `now`/`echo`) is a separate catalog-content call (flagged, not bundled).
- **Multi-catalog name resolution is a minor edge** — `callFunction` has no `catalogId`; if two registered catalogs declare the same function name with different `callableFrom`, the lookup is first-registered-match. Unambiguous for the single-catalog default; flagged for the spec if multi-catalog server-invoke becomes real.
- **Fork 5 widens `INVALID_FUNCTION_CALL` beyond the two named triggers** (adds "throwing impl") — a spec-silence call; defensible (nearest code, carries `functionCallId`, non-fatal) and consistent with ADR-0026.
- **Stale → re-verify (on ratify + build gate):** runtime SPEC §5.1/§5.2 + R14 · catalog SPEC-R5 · renderer LLD-C14 + §9 · protocol.ts · catalog.ts + catalog.json · dispatch.ts + renderer.ts · NEW call-function.ts · the round-trip test.

## Alternatives considered

- **Resolve `callFunction` args via `resolveValue`** (treat them as bindings) — rejected (fork 1). `callFunction` is surfaceless (no data model), so `{path}` has nothing to resolve against; and it conflates the envelope-RPC with the local binding surface the spec explicitly distinguishes (fact 6). Concrete literals match "args per the catalog schema."
- **A parallel server-function registry** — rejected (fork 2). The *same* registered functions serve both surfaces (fact 6); two registries drift. Share `catalogFunctions` + `catalog.functions`, gate by `callableFrom`.
- **Route `callFunction` through `functions.ts evaluate`** — rejected. `evaluate` does `@`-system dispatch + recursive arg `resolveValue` (binding semantics); a flat RPC needs neither and must not gain surface coupling. A separate thin `call-function.ts` keeps the surfaces distinct.
- **Default `callableFrom = clientOrRemote`** (permissive) — rejected — and moot: v1.0 **mandates** the default is `clientOnly` (fact 5), so this is not ours to choose. (The spec's default is also the least-authority one — a server cannot invoke a function unless explicitly opted in — and is non-breaking for existing catalogs.)
- **Gate the reject on `wantResponse`** (suppress `INVALID_FUNCTION_CALL` when fire-and-forget) — rejected (fork 4). A rejection is a protocol error the server needs regardless of whether it wanted a value; the spec's "MUST reject" is unconditional.
- **Map a throwing invoke to `VALIDATION_FAILED`** — rejected (fork 5). `VALIDATION_FAILED` requires `surfaceId` (excludes `functionCallId`), but a `callFunction` failure must carry the `functionCallId` the server correlates on; only the `INVALID_FUNCTION_CALL` arm does. (And the call *was* a function call, not a message-schema failure.)

## Amendment — the cross-catalog `callableFrom` collision semantic: most-restrictive-wins (`clientOnly` is a HARD FLOOR) · 2026-06-30

> Resolves the **"multi-catalog name resolution is a minor edge"** flag (Consequences + clause 4a's parenthetical) into a **ratified decision**, and corrects a build defect caught post-commit. Append-only; the original Decision (the `callFunction` RPC, the 5 forks, the two distinct surfaces, the activate-ADR-0031 relationship) **stands** — this pins the one flagged-open edge.

**The defect.** The #23 build (`call-function.ts`) implemented **first-allows-match** — it scanned registered catalogs in registration order and **skipped** a `clientOnly` declaration ("*another catalog may opt it in*"), taking the first catalog whose `callableFrom` permits remote — citing a **fabricated "ADR-0034 clause 4b"** (no such clause exists). This is **most-permissive** (fails OPEN), the opposite of the two-tier model's intent, and it was shaped to pass an AC1 fixture that **shadowed the default's `clientOnly` `required` with a `clientOrRemote` `required`** — a test driving a security semantic.

**The decision (ratified): most-restrictive-wins.** For a function **name** declared in more than one registered (active) catalog, the **effective `callableFrom` is the most restrictive across all of them**, and **`clientOnly` is a hard floor**:
- If **any** active catalog declares the function `clientOnly` → it is **server-un-invocable**; a `callFunction` for it is **rejected** (`INVALID_FUNCTION_CALL`). A permissive sibling does **not** loosen the guard.
- A `callFunction` is **invoked** only if the function is registered with an impl **and no** active catalog marks it `clientOnly` (i.e. every declaration is `remoteOnly`/`clientOrRemote`).
- The result is **independent of registration order** (unlike first-registered / first-allows).
- This **supersedes** clause 4a's parenthetical "first-registered-match" and the Consequences "minor edge" — those were a flagged-open edge, now decided.

**Why (grounded).** (1) **Fail-safe** — `callableFrom` is a security gate (which functions a server may invoke); a security gate MUST fail **closed**. most-restrictive-wins denies on any `clientOnly`; first-allows could be bypassed by registration order or a permissive sibling (a hole). (2) **Consistent with the two-tier "stricter overrides" model** — `registry.ts` (a project catalog "MAY intentionally shadow"/tighten) and `catalog/functions.ts` ("a project catalog may register a **stricter** override") both frame overrides as *tightening*; most-restrictive-wins honors that — a sibling can only tighten (`clientOnly` wins), never loosen. (3) It lets a project **tighten** a default-permissive function (mark it `clientOnly` to deny server-invoke) — which first-registered would **block**. (4) It corrects the build's most-permissive bug + the fabricated citation.

**Repairs (this amendment):** `a2ui/renderer/call-function.ts` (first-allows → most-restrictive-wins; remove the fabricated "clause 4b" citation) · the AC1 fixture (a **non-colliding** `clientOrRemote` name — `getScreenResolution`, not a shadowed `required`) + **NEW AC7** (the hard-floor positive control) · `a2ui-catalog SPEC-R5` (the collision-semantic line). Exec rebuilds the controller + fixture to match; re-gate.
