## Context

`ui-theme-provider` ships with `scheme`/`scale`/`density` live and `theme` as a reflected pure carrier
awaiting a `[theme='<name>']` CSS layer. The default color surface (`tokens.css`) is structurally
Ultimate Tokens output already — TKT-0087's build confirmed this directly, not just structurally:
driving UT's real engine against the repo's own checked-in 11-palette config
(`.claude/docs/brand/ultimate-tokens-modal-jazz-the-cool-blue-session/…-config.json`) reproduces 991
of the default's ~1007 `--md-sys-color-*` names byte-for-byte (oklch tonal primitives on UT's
050→950 stop grammar, `-scrim-{050…950}` alpha rows matching UT's `SCRIM_BASES`/`SCRIM_STEPS`, and
every `light-dark()` semantic role); the remaining ~16 are named, hand-authored agent-ui additions
(enumerated in the Decision below). UT's `cssPrefixOf` seam exists precisely to emit
"Material-flavoured names (`--md-sys-color-*`)" (exports.js:278-279). A theme pack is therefore NOT
a new format — it is a re-generation of the surface the repo already speaks, scoped under an
attribute.

## Decision

1. **A theme pack is one committed CSS file** at `shared/src/tokens/themes/<name>.css`, whose single
   top-level block `[theme='<name>'] { … }` re-declares the `--md-sys-color-*` surface (primitives +
   semantic roles), scheme-complete via the same `light-dark()` idiom as the default. Custom-property
   inheritance does the scoping: the block matches the `ui-theme-provider[theme]` element (or any
   themed root), overriding the `:root` default for that subtree — nesting works by cascade proximity,
   no specificity games.
2. **The default theme IS pack zero.** `tokens.css` stays the `:root` default (no `[theme]` required
   to render); `themes/<name>.css` files are strictly additive layers. `tokens/` organizes as:
   `tokens.css` (default color) · `dimensions.css` (the ADR-0140 system tier) · `base.css` (document
   base) · `themes/*.css` (packs). Exports gain `"./themes/*": "./src/tokens/themes/*"`.
3. **Ultimate Tokens generates; agent-ui wraps and gates — repos stay decoupled.** The pipeline
   consumes UT's EXPORT ARTIFACT (the oklch CSS export with `export.colorPrefix = 'md-sys-color'`),
   never imports UT's engine at RUNTIME: `node --experimental-strip-types
   shared/tools/themes/wrap-pack.ts <name> <ut-export.css>` wraps the body in `[theme='<name>']` and
   writes the pack. No palette-name adaptation table — a config's palette names pass through verbatim
   (verified 2026-07-17 against the repo's own real 11-palette config: agent-ui's "accent" family is
   UT's own curated name for that preset, not a rename agent-ui applied).
   A STANDING parity gate (`theme-packs.test.ts`) asserts every pack declares the default's CORE
   `--md-sys-color-*` set — "core" is the full default set MINUS 16 named, hand-authored roles NO
   stock UT export produces (`focus-ring` · six `neutral-tint-*` · `neutral-track{,-hover}` ·
   `primary-selected` · six raw 050/950 alpha-triple primitives — enumerated exactly, TKT-0087
   Findings; independently re-verified against a real generated pack, not just cause-#1's estimate).
   A pack omitting one of these 16 is CORRECT, not a gap: an element inside the themed subtree
   inherits it from `:root` via ordinary CSS custom-property cascade (`[theme=…]` is always a `:root`
   descendant) — no fallback mechanism needed, confirmed by the platform's own inheritance model, not
   assumed. A pack missing any OTHER default name, or flattening a role the default pairs via
   `light-dark()`, is a red gate — both checks carry a genuine negative control proving they bite.
   Per-pack AA/contrast remains the generator's discipline (UT's own ds-gates) + the repo's existing
   color-audit process; the parity gate is structural, not perceptual.
4. **On-demand loading is the APP's job; the component stays inert.** `ui-theme-provider` is untouched
   (its LLD's own prediction). A small site helper (`site/lib/theme-loader.ts`) lazy-injects a pack's
   stylesheet the first time that theme is selected (`import('…themes/<name>.css?url')` →
   `<link rel=stylesheet>`), sets `theme` on the shell's provider, and persists the choice — the same
   DEV/app-owns-delivery discipline as every live seam (the packaged component carries no fetch/env
   code, the ADR-0131 pattern).
5. **The site shell dogfoods the whole system.** The doc-page shell (and the standalone app pages)
   mounts ONE `ui-theme-provider` around its content with a shell-header control: the scheme toggle
   (auto/light/dark) and the theme picker (default + packs), both persisted. Ship the default plus
   at least TWO UT-generated proof packs so the seam, the parity gate, and the loader are all
   exercised by real artifacts.

## Consequences

- The reserved seam becomes real with zero component change — the strongest validation of the
  original fence design.
- Theme switching cost = one stylesheet injection + attribute flip; unselected packs cost zero bytes.
- The parity gate makes pack drift (a role added to the default but missing from a pack) a standing
  red, which is what keeps N packs honest as the role surface evolves.
- Packs are color-only in v1; a dimensions/type pack (UT exports those too) is a named follow-up on
  the SAME `[theme]` surface, unblocked by ADR-0140's single namespace.
- The pack pipeline depends on UT's export grammar staying stable; the wrap tool validates its input
  shape and fails loudly on drift (never a silently wrong pack).
- **A real prerequisite the build surfaced, not foreseen at proposal time:** driving UT's actual
  engine against the repo's real 11-palette config found the committed `tokens.css` used the raw
  alpha-scrim primitive's PRE-ADR-016 name (`{family}-500-{step}`), while UT's `main` had since
  renamed it (`{family}-scrim-{step}`) as part of retiring its own old/new-grammar split repo-wide.
  Adopting the same name here (a small, separately-committed migration — TKT-0087 Findings) was the
  right call over carrying two grammars forward, per the sibling repo's own recommendation.
