## Context

ADR-0027 shipped `${…}` DynamicString **path** interpolation and built the scanner/classifier **forward-compatible**: the classifier's `(`-bearing branch (`interpolate.ts` ~line 127) was carved out as the deferred **function-expression** form and rendered **verbatim** (`out += '${' + body + '}'`), with the seam designed so "this single branch is extended to parse `body` into a FunctionCall and route it to `evaluate()` — no scanner change required." Task #15 is that extension.

Grounded against the repo at the ADR-0027 build (gate-green) and the **A2UI v1.0** expression-grammar facts fetched from a2ui.org and supplied by team-lead (Constraint **C1** — conform, cited-not-redefined). The grammar:

- An expression body containing `(` is a **function-expression**: `name(arg:value, arg2:value2)` — a UAX-31 identifier, parentheses, and a **named** argument list (`arg:`).
- An argument **value** is one of: a **quoted string literal** (`'yyyy-MM-dd'`); a **nested `${…}` path** (`value:${/currentDate}` — a JSON-Pointer reference); a **number/boolean literal**.
- The two canonical examples: `${now()}` (no-arg) and `${formatDate(value:${/currentDate}, format:'yyyy-MM-dd')}` (string-literal + nested-path args).

The repo already has the resolution engine for this: `functions.ts evaluate(call, surface, itemScope, emitError, registry)` (ADR-0026) dispatches a `FunctionCall` on namespace (`@`-system vs catalog), resolves each named arg **recursively** through `resolveValue`, runs the impl, and emits `FUNCTION` on unknown/throw. So #15 is a **parser**, not a second evaluator: it turns the function-expression *string* into a `FunctionCall` *object* and hands it to `evaluate`. Dispatch, recursion, reactivity (SPEC-N2), and the FUNCTION error are all reused.

**Grounding pinned (team-lead fetched the v1.0 function-call grammar section).** Three facts settle the design:
- **Dispatch — `@` is the ONLY system namespace.** *"Function names beginning with `@` represent universal system context evaluations available across all catalogs. Custom catalogs MUST NOT define functions prefixed with `@`."* `now`/`formatDate`/`formatNumber`/`formatString`/`pluralize`/`required`/`regex`/`length`/`email`/`and`/`or`/`not`/… are all **catalog-function examples**, **not** built-ins — there is **no** `now()` system function. So **#15 is purely a parser; it adds zero system functions.** A bare `name(...)` parses to `{call: name}` (no `@`) → the **catalog** registry; against the default catalog (`required`/`email`/`regex` only) `${now()}` correctly emits `FUNCTION` until a project catalog registers `now`. One wrinkle: the parser MUST also accept an **`@`-prefixed name with parens** (`@index(offset:1)`) and route it to `evaluate`, which already dispatches `@`-vs-catalog and handles `@index`+`offset` — so both `@name(args)` and `name(args)` flow to the same `evaluate`.
- **Arg syntax — named args; `${…}`-wrapped paths; quoted-string/number/boolean literals.** *"named arguments only"*; an arg **value** is a quoted string literal (single-quote shown, `'yyyy-MM-dd'`), a number, a boolean, **or a nested `${…}` expression** — a path **must be `${…}`-wrapped** (`value:${/currentDate}` ✓, `value:${firstName}` ✓; a **bare** `value:firstName` is **not** allowed). Nested expressions as arg values are explicitly permitted (`${upper(${now()})}`). One outlier: that nesting example uses a **single positional** arg (`upper(${now()})`) — the only positional form the spec shows.
- **Malformed — spec-silent, deferred to the client.** *"The spec does not define error handling for malformed function-call expressions, unknown functions, or function exceptions… defers to the client implementation."* So the error model below is a documented **design decision**, not a spec mandate.

**Design insight (pins the whole resolution + reactivity story): the parser translates the expression grammar INTO the existing JSON value shapes `resolveValue` already dispatches — `literal | {path} | {call}`.** `${formatDate(value:${/currentDate}, format:'yyyy-MM-dd')}` parses to `{ call:'formatDate', args:{ value:{path:'/currentDate'}, format:'yyyy-MM-dd' } }`. Then `evaluate` does **all** resolution: it already recursively resolves each named arg via `resolveValue`, so a `{path}` arg resolves through the per-path memo **inside the bound-prop effect** → reactivity is **free**, and a nested `${call(...)}` arg → a nested `{call}` object → `resolveValue` recursion (also free). **The parser is pure syntax — string → `FunctionCall` whose arg values are `literal | {path} | {call}` — adding zero evaluation/resolution/reactivity logic.**

## Decision

We deliver the function-expression arm in a new **`renderer/fn-expr.ts`** (a small tokenizer/parser, pure, zero-dep) and swap the `interpolate.ts` classifier `(`-branch from verbatim-render to **parse → `FunctionCall` → `evaluate` → coerce → concat**. The scanner, the path arm, and ADR-0027's render-literally model are unchanged.

1. **`parseFunctionExpr(body) → FunctionCall | null`.** Tokenize a function-expression body: the leading **call name** (a UAX-31 identifier, **optionally `@`-prefixed** so `@index(offset:1)` parses) up to `(`; the **named argument list** inside the balanced `(…)`, split on **top-level** commas; each arg `name:value` split on its **first top-level** `:`. An **arg value** is parsed by leading token: a quoted `'…'`/`"…"` → a **string literal**; a number/`true`/`false` token → a **number/boolean literal**; a `${…}` → recurse the inner body through the **same classifier** (`${/p}` → a `{path}` Binding; `${fn(…)}` → a nested `{call}` Binding via a recursive `parseFunctionExpr`). A **bare** identifier value (`value:firstName`, not `${…}`-wrapped) is **invalid** per the grammar — it is **not** a path; the parser rejects it (→ `null` → verbatim). Whitespace around the name, each arg, the `:`, and each value is trimmed. **Top-level** = not inside a nested `(…)` or `${…}` (depth-tracked), so a comma/colon inside a nested arg is not a delimiter. Returns `null` on any tokenize failure (malformed) **or** a **positional** arg (see Fork 1).
2. **Parse INTO the existing JSON value shapes — the parser adds no resolution.** The parsed args become `FunctionCall.args: Record<string, Binding>` whose values are exactly `literal | {path} | {call}` — the shapes `resolveValue` already dispatches. A `${path}` arg → `{ path }`; a string/number/boolean → that **literal**; a nested `${fn(…)}` arg → a nested `{ call, args }`. `evaluate` then does **all** resolution via its existing per-arg `resolveValue` recursion: a `{path}` arg reads the per-path memo, a nested `{call}` re-evaluates, a literal returns as-is. **Zero new resolution/reactivity code** — the parser is pure syntax.
3. **Route to the existing evaluator + coerce.** The classifier branch becomes: `const call = parseFunctionExpr(body); if (call) out += coerce(evaluate(call, surface, itemScope, emitError, registry)); else out += '${' + body + '}'`. `evaluate`'s result is coerced through the **same** `coerce()` table ADR-0027 defined (number/boolean → `String`; null/undefined → `""`; object/array → `JSON.stringify`; string → itself) — so a function returning `undefined` (the FUNCTION-error path) splices `""`, consistent with an unresolved path.
4. **Nesting + reactivity are free (the parse-into-JSON insight).** A `${/path}` arg is a `{path}` Binding inside the `FunctionCall`; a nested `${fn(…)}` arg is a nested `{call}` Binding. `evaluate` resolves each via its existing recursive `resolveValue → resolve`/`evaluate` (the per-path memo / the evaluator). Because `interpolate` runs **inside the scope-owned bound-prop effect** (`widget.ts`), every embedded `${/p}` read subscribes the effect to that path — so a fn-expr embedding `${/p}` (top-level arg **or** inside a nested call) **wakes on `/p` change** (SPEC-N2, reused unchanged; the "Text updates as data changes" guarantee extended to function-expression templates). No special nesting handling — the JSON `{call}`/`{path}` recursion already does it.
5. **Seam signature.** `interpolate(s, surface, itemScope, resolve)` gains `emitError` + `registry` → `interpolate(s, surface, itemScope, resolve, emitError, registry)`, threaded from the `resolveValue` call site in `functions.ts` (both are already `resolveValue` params and in scope). The path arm does not use them; only the function-expression branch does.

### Forks settled

- **Fork 1 — POSITIONAL args DEFERRED; named args (with nested `${path}`/`${call}` values) supported.** v1.0 is *"named arguments only"* and `FunctionCall.args`/`FunctionDef.args`/`evaluate` are all **named** `Record<string, Binding>` (ADR-0026). The one positional form the spec shows — the nesting outlier `${upper(${now()})}` (a single **unnamed** arg) — has no name to key onto and would require either inventing a positional→named convention the spec does not define or widening `FunctionCall` to carry positional args (a protocol change rippling through `evaluate` + every `FunctionDef`). Neither is warranted for one example using a function (`upper`) absent from the default catalog. So the parser returns `null` for a positional arg → that segment renders **verbatim** (never a wrong evaluation); booked as an additive follow-up. **Named nested calls are NOT deferred** — `outer(x:${inner(y:'z')})` parses to a nested `{call}` arg for free via clause 2's parse-into-JSON, so the only deferral is the *positional/unnamed* form.
- **Fork 2 — error model: parse-success-but-unknown/throwing-fn → `FUNCTION` + `""`; malformed-parse → verbatim, no code.** The spec is **silent** on all three (malformed, unknown fn, fn exception — *"defers to the client implementation"*), so this is a documented design decision. A successfully-parsed call is a **real evaluation**, so it rides `evaluate`'s existing `FUNCTION` contract (unknown name, missing impl, or throw → `emitError(FUNCTION)` + `undefined` → coerces to `""`). This **partially overturns** ADR-0027's *"reuse FUNCTION rejected"* alternative — that rejection was explicitly conditioned on a segment the interpolator *"does not yet parse… unimplemented, not unknown."* Once #15 parses it, it **is** an attempted evaluation against the catalog, so `FUNCTION` is correct (the server learns its catalog function is missing — the right signal). A body that does **not** tokenize (malformed grammar — unbalanced quote, bad arg syntax, a bare-identifier or positional arg) keeps ADR-0027's **render-verbatim, no error code** (graceful degradation, SPEC-N4/R4 AC2).
- **Fork 3 — whitespace + 1:1 named-arg mapping + `@`-name routing.** `name( arg : 'v' )` is trimmed at every boundary; parsed names map 1:1 onto `evaluate`'s `args` object (already `Record<string, Binding>`) — no renaming. The call **name** may be `@`-prefixed (`@index(offset:1)`); the parser preserves the `@` and `evaluate` dispatches it to the system table (the only system fn is `@index`, which already reads `offset`). A bare catalog `name(...)` dispatches to the catalog registry. No system-table expansion — both name forms flow to the **one** `evaluate`.
- **Fork 4 — paths MUST be `${…}`-wrapped; nesting via the JSON `{path}`/`{call}` recursion (clause 4).** A path arg is `value:${/d}` or `value:${name}` (relative-in-scope); a **bare** `value:name` is invalid (corrected from an earlier "bare-path-as-relative" lean — the spec shows paths only inside `${…}`). The inner `${/d}` is parsed to a `{path}` Binding and resolved by `evaluate`'s recursion through the per-path memo, so reactivity holds without new machinery. Quote leniency: both `'…'` and `"…"` delimit a string literal (the spec uses `'…'`; the as-built #13 test fixture uses `"…"` — accepting both is Postel-safe and free).

## Consequences

- **The `(`-classified branch now resolves; the #13 verbatim tests FLIP.** `interpolate.test.ts` proof-point-6 (`${now()}` → `'${now()}'`; `${formatDate(value:${/d}, format:"yyyy")}` → verbatim) asserted the *deferral*; they now assert *resolution* (`${now()}` against a catalog without `now` → `""` + a FUNCTION error; `${formatDate(...)}` against a catalog with `formatDate` → the formatted string). These are the contract-flip tests the build must rewrite — flagged so the build does not preserve a now-stale assertion.
- **No second evaluator, no new system functions, no new error code.** The grammar parses **into** ADR-0026's `evaluate`; dispatch, recursion, FUNCTION, and reactivity are reused. `fn-expr.ts` is pure parsing; the only behavioral change in `functions.ts`/`interpolate.ts` is threading two existing deps and swapping one branch.
- **The scanner and the path arm are byte-for-byte unchanged.** ADR-0027's escape-aware scanner already captures a nested-`${}` arg whole (verified: `interpolate.test.ts:244` shows the outer `(` body includes the inner `${/d}`), and brace-depth keeps it parse-stable — so #15 touches only the classifier branch, exactly as ADR-0027's seam promised.
- **`${now()}` against the default catalog renders `""` + FUNCTION, not a date.** #15 adds no display functions; the default catalog ships validators only. This is correct conformance (the function is genuinely absent), and the right place to add `now`/`formatDate` is a catalog concern (a project catalog, or a future default-catalog display-function slice — out of scope here, notable as a UX gap if a demo expects `${now()}` to render a date).
- **Named nested calls work; only POSITIONAL args are deferred.** `outer(x:${inner(y:'z')})` parses to a nested `{call}` arg and evaluates for free (clause 2 parse-into-JSON). The one deferral is the **positional/unnamed** form (the `${upper(${now()})}` outlier) — it renders verbatim until a follow-up (booked below); the spec's mainline examples are all named and resolve.
- **Stale → re-verify (on ratify + build gate):** runtime SPEC-R10 (function-expression clause) · renderer LLD-C10 §7 (classifier `(`-branch resolves) · the new `fn-expr.ts` · `interpolate.ts` (branch swap + signature) · `functions.ts` (`resolveValue` threads deps) · `interpolate.test.ts` proof-point-6 (flipped).

## Alternatives considered

- **Fold the function-expression parser into the existing scanner (one pass, no separate `fn-expr.ts`)** — rejected. The scanner is a character-level segment splitter; the function-expression grammar is a token-level recursive parser (identifier, balanced parens, named args, typed values). Conflating them would re-entangle the two concerns ADR-0027 deliberately separated (scan vs classify+resolve) and make the deferred-nested-call boundary harder to draw. A small dedicated `fn-expr.ts` keeps the scanner pristine and the grammar testable in isolation.
- **Add `now`/`formatDate` as SYSTEM functions (`@now`, or a standard-library table)** — rejected. v1.0 reserves `@` for context like `@index` (`naming.ts`), not a function standard library; the two examples are catalog-function *examples*, not system functions. Minting `@now`/`@formatDate` would invent system semantics the spec does not grant (Constraint C1) and bypass the catalog allowlist (SPEC-R9). Display functions belong in a catalog; the parser stays catalog-agnostic.
- **Parse the positional/unnamed arg form (`${upper(${now()})}`) this wave** — rejected, deferred. v1.0 is "named arguments only" except this single nesting outlier; `FunctionCall.args`/`FunctionDef.args`/`evaluate` are all named `Record` (ADR-0026). Supporting it needs either a positional→named convention the spec does not define or a `FunctionCall` protocol widening rippling through `evaluate` + every `FunctionDef` — disproportionate for one example using a non-default-catalog function. The parser returns `null` for it (verbatim, never a wrong evaluation); booked as a follow-up. **Named** nested calls are supported (not part of this rejection).
- **Emit a new `INTERPOLATION`/`EXPRESSION` error code for a malformed function-expression** — rejected, same reasoning as ADR-0027: the spec defines no parse-error code, the only render-time code is `FUNCTION` (for an *evaluated* call), and render-verbatim is graceful and reversible. A malformed body is most plausibly authoring text; tearing it out is worse than showing it.
- **Resolve a nested `${/path}` arg by pre-interpolating the arg string, then passing a resolved literal to `evaluate`** — rejected. That would coerce the path value to a **string** before the function sees it (losing number/object types a function may need) and would resolve the path **outside** `evaluate`'s recursion, duplicating the resolver. Passing a `{path}` Binding lets `evaluate` resolve it with the correct type through the existing per-path memo — one resolution path, type-preserving, reactive.
