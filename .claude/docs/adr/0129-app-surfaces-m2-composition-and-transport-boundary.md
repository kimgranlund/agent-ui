## Context

M2 is the milestone `agent-app-surfaces.prd.md` §7 named but never specced: **PRD-G3** (agent-emitted content appears with zero app-written renderer/transport glue) and **PRD-G4** (a multi-turn conversation presents from a primitive, not hand-built chrome). Unusually for a design intake, M2 does not start from a blank page — two independent site-level prototypes already prove the mechanism works:

- `site/pages/a2ui-live.ts` + `site/lib/canvas-surface.ts` — a persistent, translate-centered artboard the page hand-wires directly to a `createRenderer()` `RendererHost`, plus (ADR-0097 §2) an inline per-"ask" mount registry (`site/lib/ask-registry.ts`).
- `site/pages/a2ui-chat.ts` + `site/lib/surface-registry.ts` (TKT-0020, shipped 2026-07-11, six days AFTER the PRD's own ratification) — a single scrolling log where EVERY surface a turn creates gets its own persistent, inline mount, generalizing `ask-registry.ts`'s "asks only" lifecycle to "every surface."

Both are real, gate-green, browser-proven code. The design questions this ADR resolves are not "can this work" (proven) but **where the primitive/app boundary falls**, given both PRD-D2 (host chrome is the trusted frame; agent content lands only inside a bounded mount) and PRD-D3 (the DAG-forcing cycle argument already put this tier in `@agent-ui/app`, above `@agent-ui/a2ui`) are already ratified and not re-litigated here.

## Decision

Realized in four clauses; the M2 SPEC/LLD own the concrete mechanism.

1. **`ui-surface-host` owns ONLY the mount + stream — never a transport, a produce loop, or provider config.** It wraps exactly one `RendererHost` (mount at connect, `ingest`/`finalize`/`dispose`, an `onClientMessage` callback) and exposes no transport/provider-shaped type. This resolves the PRD's own §5 open question ("does the canvas/surface-host fully own the `produce` loop... or only mount + stream") as **only mount + stream** — the model-calling machinery (`packages/agent-ui/a2ui/tools/agent/produce.ts`) stays Node/site-scoped, never promoted into a package, matching its existing SPEC-N1 posture (not a package export today, and this ADR does not change that).
2. **`ui-conversation` composes `ui-surface-host` internally, one per open surface, generalizing `SurfaceRegistry`/`AskRegistry` as its OWN mechanism.** A fresh `surfaceId` mounts a new `ui-surface-host` inline in that turn's bubble; a known `surfaceId` routes to its original host/bubble (persistent identity across turns); `deleteSurface` disposes that one host with a visible, non-removable annotation. This composition is now edge-to-edge sound because the accepted ADR-0128 makes a resent, already-mounted container's record reconcile correctly — a precondition this ADR's mechanism assumes, not re-derives.
3. **Wire disclosure is opt-in (default off); narration is not.** `ui-conversation` ships ADR-0088's honest per-turn narration (composing the shipped `ui-status-stream`) unconditionally — hiding what the agent is doing would contradict the fleet's own transparency law. The raw-JSONL `<details>` wire dump `a2ui-chat.ts` also ships today is a debugging/inspection affordance, gated behind a `disclosure: boolean` prop, default `false`.
4. **Both existing site chat surfaces migrate onto the primitives, and their generalization sources are deleted.** `a2ui-live`'s persistent canvas + chat pane + inline asks re-express on `ui-surface-host`/`ui-conversation` (the PRD's own literal M2 target); `a2ui-chat.ts` — a newer, more complete implementation of the identical lifecycle, shipped after the PRD but before this intake — ALSO re-expresses on `ui-conversation` in full. `site/lib/canvas-surface.ts`, `surface-registry.ts`, and `ask-registry.ts` are then deleted; carrying two independent hand-built implementations of a lifecycle the shipped primitive now owns would itself violate PRD-G6's coherence bar.

### Forks for Kim (each with a firm recommendation; the recommendation is the default absent an objection)

- **F1 — the transport/produce-loop boundary.** *Recommend: the imperative-handle design in clause 1* — `ui-surface-host`/`ui-conversation` expose zero transport-shaped API; the app's own turn loop calls `ingest()`/`AgentTurnHandle.ingestLine()` as its own `AgentTransport.turn()` yields lines. Alternative considered: a shared `TransportSeam` interface the conversation primitive accepts as a constructor/prop value (a narrowed re-export of `AgentTransport`'s shape) — rejected because it still couples the primitive's public surface to a transport *shape*, when PRD-D2's trusted-frame law argues the primitives should only need to agree on the WIRE format (`A2uiServerMessage`/`A2uiClientMessage`, already a public spec-owned type), never on how a turn is produced or authenticated.

- **F2 — composition: does `ui-conversation` own `ui-surface-host` internally, or does the app compose both separately?** *Recommend: internal composition (clause 2)* — this is what makes PRD-G4's "0 hand-built thread" claim literally true. Alternative considered: `ui-conversation` renders only text bubbles, leaving every surface mount to app-authored code keyed by the app's own bookkeeping — rejected because it re-creates exactly the per-app registry/routing logic (`surface-registry.ts`) this milestone exists to delete; the primitive would narrate but not host.

- **F3 — narration vs. wire disclosure: both mandatory, both optional, or split?** *Recommend: split (clause 3)* — narration mandatory (a transparency LAW, ADR-0088), wire disclosure optional (a debugging affordance, not every consumer's product surface should show raw JSON by default). Alternative considered: both mandatory (today's `a2ui-chat.ts` shape exactly) — rejected as presuming every consumer wants a permanent raw-wire panel, which is a developer-tool decision, not a chat-UX one. Alternative considered: both optional — rejected because narration silence would contradict the fleet's own honesty posture; it is not this primitive's call to make optional.

- **F4 — reference-app scope: `a2ui-live` only (the PRD's literal wording), or `a2ui-live` + `a2ui-chat`?** *Recommend: both (clause 4)* — `a2ui-chat.ts` shipped six days after the PRD's ratification and independently reinvented (a strict superset of) the same per-surface lifecycle mechanism this milestone promotes. Leaving it unmigrated is a foreseeable, immediate coherence regression the moment `ui-conversation` ships. Alternative considered: `a2ui-live` only, treating `a2ui-chat.ts`'s migration as a separate future ticket — rejected as needlessly deferring a fix to a problem this same intake already discovered and can close in the same wave.

## Consequences

- **`@agent-ui/a2ui` moves from a declared-but-unexercised M1 dependency to a genuinely imported one** — the layering trip-wire's negative control must still prove the apex stays un-imported from below; no new DAG node, no cycle.
- **`@agent-ui/app`'s size budget re-bases again** (M1: ≤~3 KB provisional → measured 24829 B gz within 26624 B gz after M1+M4; M2 adds two more elements against ~1.8 KB of remaining headroom) — the ceiling is set against a measured post-M2 tree, not guessed in advance (the M1/M4 kickoff discipline, unchanged).
- **Two site-land modules retire** (`surface-registry.ts`, `ask-registry.ts`) **plus one artboard helper** (`canvas-surface.ts`) — their mechanism does not disappear, it is promoted; anything that imported them directly (only the two chat pages, verified) must land its own migration in the same wave the deletion commits, per the LLD's ordering guard.
- **The wire-disclosure feature moves from "every consumer gets it" to "an opt-in prop"** — a future consumer wanting a permanent debug panel still can (the prop stays supported), but the shipped default product experience no longer shows raw JSONL.
- **Stale → re-verify at the M2 build wave:** `agent-app-surfaces.prd.md` §5's "still open" sub-questions (produce-loop ownership, the conversation+canvas-host pairing) are now answered by this ADR and should be marked resolved, not re-opened · the PRD-G3/G4 metric rows read against this SPEC's ACs · `CLAUDE.md`'s `app/` line if `ui-surface-host`/`ui-conversation` earn a mention.

## Alternatives considered

- **A single combined `ui-agent-surface` element (conversation + canvas fused into one tag).** Rejected: it cannot serve `a2ui-live`'s shape (a persistent, chat-external canvas) and `a2ui-chat`'s shape (every surface inline in the log) from one API without a mode axis that would itself need this ADR's same forks answered first — decomposing into two composable primitives is strictly more general and is what the PRD's own goal IDs (separate PRD-G3/PRD-G4) already imply.
- **Placing `ui-surface-host` in `@agent-ui/a2ui` itself (the PRD's Fork 2/PRD-D2 territory revisited).** Rejected: already settled at the PRD level — the DAG-forcing cycle argument (PRD-D3, Fork 3) requires anything importing `a2ui`'s renderer to live ABOVE it; `@agent-ui/app`'s M1 `package.json` already declared this dependency for exactly this purpose. Re-litigating it here would contradict a ratified decision this intake was told not to re-open.
- **A synchronous `render(lines: string[])` batch API instead of `ingest(line)` streaming.** Rejected: A2UI's whole value proposition is progressive rendering as lines arrive (`a2ui-runtime.spec.md` SPEC-N4); a batch API would force the app to buffer a full turn before any paint, regressing the exact behavior `a2ui-live`/`a2ui-chat` already prove works well streamed.
