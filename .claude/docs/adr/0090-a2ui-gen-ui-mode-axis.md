## Context

ADR-0089 (`proposed`, 2026-07-07) taught the live agent to ASK — two behaviors in the hand-authored
GRAMMAR half of the derived prompt: **clarify-when-underdetermined** and **catalog-boundary negotiated
approximation**. They shipped **unconditional**. `buildSystemPrompt(catalog, exemplars)` takes only a
catalog and exemplars (`system-prompt.ts:106`); the GRAMMAR is a single `const` string
(`system-prompt.ts:26-73`) concatenated on every call (`:107-114`), so **every live turn, in every
deployment, gets the identical disposition** — the clarify paragraph (`system-prompt.ts:38-44`) and the
boundary/negotiate paragraph (`system-prompt.ts:46-54`) fire the same way whether the catalog is a
three-artifact curated set or the whole fleet. A grep of the runtime confirms **no mode/flag concept
exists** anywhere in `tools/agent/*` or `site/lib/*` today.

Kim named **three distinct Gen-UI use cases** that pull that one grammar in different directions:

- **Specific, actionable Gen UI** — a constrained catalog-based chatbot rendering a few curated
  artifacts fast. It wants the ASK grammar dialed DOWN: most requests should map directly to a catalog
  artifact, and stopping to ask "better in what way?" or negotiating "I can approximate a data-table with
  a Grid of Rows — want me to?" is friction, not help. A curated deployment should prefer directive
  mapping and, at its wall, decline-and-redirect to what it actually offers.
- **Blue-sky Gen UI** — an open-ended reasoner ("make a card-game UI component sheet"). It wants the ASK
  grammar dialed UP: lower the clarify threshold, welcome several clarifying rounds, compose more
  elaborate approximations, and narrate its reasoning.
- **Structural Gen UI** — "the browser loads hardcoded JSON structures that were generated by an LLM at
  some point, and simply renders them." Verified against the tree, this is **not a live-grammar mode at
  all**: `createRecordedTransport(transcript)` (`recorded-transport.ts:25-36`) already replays ANY
  committed `RecordedTranscript`'s `lines` through the SAME `AgentTransport` seam the page consumes
  (`a2ui-live.ts:156`, SPEC-R1 AC2), with zero network, zero key, and **no live model** — and validity is
  provable at commit time by the existing `validateA2ui`/`examples.test.ts` gate (`transcript.ts:2-3`;
  the same shared validator `produce.ts:19` uses). So Structural is the **already-shipped recorded
  transport**, not a prompt disposition; it needs zero new runtime mechanism, only a doc/example
  clarifying it is a first-class pattern.

Two shipped facts bound any config axis, and both keep it prompt-level, not wire-level:

- **The disposition lives entirely in the GRAMMAR half.** ADR-0071 permits the hand-authored grammar to
  carry non-derived instruction; the note channel (ADR-0088) and the ASK behaviors (ADR-0089) both ride
  it, and `prompt-drift.test.ts` asserts only the catalog-DERIVED inventory (`catalogInventory`,
  `system-prompt.ts:75-84`, emitted under the `## Available components` header at `:109`) as SET-EQUAL.
  A mode axis that touches ONLY the grammar half therefore adds no wire, transport, or protocol surface
  and leaves the drift gate untouched — the cheapest possible seam.
- **The catalog boundary is a HARD security allowlist (SPEC-R9), not a disposition.** An uncatalogued
  type fails the shared validator and never streams (`conformance.ts:18-19`; the shared validator is the
  gate at `produce.ts:206`, and an invalid payload halts at `produce.ts:217` rather than streaming) — the
  exact mechanism ADR-0089 closed the "improvise" trap on. So "dial down the negotiation" can NEVER mean
  "let a specific-mode agent emit new types," and "dial up blue-sky" can NEVER mean "widen the allowlist."
  The honesty floor — never invent a type/prop, never pass a wrong thing off as real — must hold in EVERY
  mode; only the *exploratory* behaviors (whether to ask, whether to propose composing an approximation)
  are what a mode may scale.

The gap Kim names: today the grammar is one-size-fits-all, so a deployment gets neither the directive
chatbot nor the exploratory reasoner — everyone gets the balanced middle. There is no place to say which.

## Decision

**Introduce a Gen-UI `mode` axis: a per-turn `GenUiMode` value threaded from the proxy request body
through `produce()`'s `ProduceOptions` into `buildSystemPrompt`, that SCALES the ADR-0089 clarify/negotiate
grammar between a directive `specific` disposition and an exploratory `blue-sky` disposition — with a
default that reproduces today's ADR-0089 grammar byte-for-byte.** Four coupled clauses; they are ONE
decision because they all answer the single question *"how should Gen-UI disposition be configured, and
what may a mode touch?"* and each clause bounds the others — the floor (§2) bounds what a mode may change,
Structural (§3) bounds what is on the axis, and the surface (§4) bounds where a mode is set. Bundled as
ADR-0088/0089 bundled their coupled clauses; ratify all four knowingly.

### 1. The mode axis — a per-turn `GenUiMode` that scales the ADR-0089 grammar.

The GRAMMAR half is decomposed into an INVARIANT spine (the note-line instruction, the honesty floor
§2, the JSONL output rules) plus a MODE-SCALED block (the clarify paragraph `system-prompt.ts:38-44`
and the negotiated-approximation paragraph `system-prompt.ts:46-54`). `buildSystemPrompt(catalog,
exemplars, mode)` composes the spine + the mode's variant of the scaled block:

- **`specific`** — directive. Clarify is dialed DOWN: prefer mapping a request to the nearest catalog
  artifact and act; on a genuinely unmappable or ambiguous request, prefer a brief "I can show you X or
  Y" (decline-and-redirect to the curated set) over an open "what do you mean?". Negotiated approximation
  is dialed DOWN: at the wall the agent names the limit and points to what it *does* offer rather than
  proposing to compose a novel primitive-built approximation.
- **`blue-sky`** — exploratory. Clarify is dialed UP: a lower threshold, several clarifying rounds
  welcome (each a note-only turn, each answer a normal `intent` turn — ADR-0088 §2 session history
  already connects them, so N rounds work with no new mechanism), and clarifying even when a sensible
  default exists is encouraged. Negotiated approximation is dialed UP: compose more elaborate
  approximations, iterate, and use the `note` channel to narrate the reasoning ("I considered a Grid vs a
  stacked Column; going with Grid because…") — the note is shown verbatim today (`a2ui-live.ts:240`), so
  the reasoning trace rides ADR-0088's existing channel with no extension. Blue-sky additionally
  instructs a **two-direction composition discipline** before it emits: reason **top-down** from the
  user's goal to the parts a UI for it needs (a card-game sheet needs a hand display, a discard pile, a
  score readout, an action bar); then **bottom-up** from the catalog to which real primitives realize
  each part (a hand ≈ a `Row` of `Card`s, a score ≈ a `Grid` of `Text`, an action bar ≈ a `Row` of
  `Button`s); then **reconcile** — build every part a primitive hosts, keep the surface minimal (add no
  structure the goal does not require — the shipped "Keep the surface minimal and correct" rule,
  `system-prompt.ts:73`), and for any part the catalog cannot host, do NOT invent: fall through to the §2
  honesty floor / ADR-0089 catalog-wall behavior (name the limit in the `note`, approximate within the
  catalog, disclose). This is a right-sized, prompt-level analog of the `/system-decompose` two-plane
  method whose cross-check quadrant maps one-to-one onto behaviors this system already has: a part with a
  hosting primitive is built, a part with none is the catalog wall ADR-0089 already handles, and a
  primitive with no motivating part is the gold-plating the minimal-surface rule already forbids. It
  therefore adds **no new mechanism** — it is reasoning prose in the mode-scaled grammar block, and its
  "I considered a Grid vs a stacked Column…" trace rides blue-sky's already-dialed-up `note` narration
  (MAY narrate, never MUST dump a decomposition as JSONL).
- **default** — the balanced middle: the composition, at the default mode, emits the EXACT ADR-0089
  grammar string, byte-for-byte. Absent `mode` ⇒ default ⇒ zero regression (the ADR-0088 opt-out-default
  discipline: absent means today's behavior).

The two-direction discipline is **blue-sky-only and additive**: `specific` mode deliberately does not
decompose (it prefers a direct catalog-artifact mapping — decomposition there is friction), and the
**default composition stays byte-identical** (the enrichment touches only blue-sky's variant of the
scaled block, so `prompt-drift.test.ts` and the default byte-identity assertion are unaffected). To
supply the *catalog-specific* half of the reasoning (which primitives realize which idiom, and where the
walls are — knowledge a general model cannot have about this 32-component catalog), the blue-sky grammar
carries a small set of hand-written **calibration examples** in the *need → parts → catalog mapping →
what falls to the wall* shape, grounded in the post-ADR-0087 whole-fleet catalog; the indicative set (a
card-game sheet, a settings screen, a dashboard, a form, a master-detail split) is fixed at build, with
~2–3 inline and the remainder seeded as retrievable corpus exemplars to bound per-turn token cost.
Blue-sky is precisely the regime where `retrieve()` is weakest (open-ended prompts share little
vocabulary with the corpus and often return `[]`, `retrieve.ts:92-99`), which is *why* the structural
reasoning instruction — not exemplar imitation alone — is blue-sky's primary composition aid. (A live
model mid-generation has no Skill-tool path — `produce.ts:152-153,177-182` shows it sees only the
system-prompt string and retrieved exemplars — so this reasoning discipline and any pattern knowledge
must ride one of those two channels, never a Skill invocation; `/ui-patterns` and `/system-decompose`
inform this design's SHAPE, they are not consulted at runtime.)

The exact mode names/count and the precise per-mode wording are build-tuned (see *re-verify*); `specific`
/ `blue-sky` / default are the indicative set.

### 2. The honesty floor is mode-INVARIANT — a mode never touches the SPEC-R9 security allowlist.

The "NEVER invent a component or prop … never silently substitute something else and pass it off as the
real thing" invariant (`system-prompt.ts:47-48,53-54`, the SPEC-R9 allowlist, enforced by the shared
validator at `produce.ts:206` which halts at `produce.ts:217`) is
lifted OUT of the mode-scaled block into the invariant spine. Every mode — `specific`, `blue-sky`,
default — carries it identically. A mode scales *whether the agent asks / whether it proposes an
approximation*; it can never license emitting an uncatalogued type, and it cannot make the agent pass off
a wrong-but-valid surface as the real thing. This is the clause that keeps "dial down the negotiation"
from ever becoming a security weakening.

### 3. Structural is the recorded transport, NOT a grammar mode — a doc clarification, zero new mechanism.

Structural Gen UI is realized by the SHIPPED `createRecordedTransport` path (Context): a committed
`RecordedTranscript` of pre-generated JSONL, validated at commit time by the existing
`validateA2ui`/`examples.test.ts` gate, rendered through the SAME `createRenderer()` host the live path
uses — with no live model in the loop. The deliverable is a **doc + a worked transcript example** naming
this as a first-class pattern (LLD-C2), plus the standing validity gate on that example (the
`round-trip.test.ts`/`examples.test.ts` precedent). It does NOT sit on the `GenUiMode` axis: that axis
reaches `buildSystemPrompt` (the LIVE path), whereas Structural means "do not call the live path at all"
— a transport choice at a different layer. Conflating the two would put a value on the enum that the
prompt composer must special-case to "skip myself," a category error. (Whether to nonetheless expose a
uniform three-value use-case surface for demo legibility is the one open fork — see below.)

### 4. The configuration surface — `ProduceOptions.mode` (core) + a dev switcher (demo) + a construction-site default (deployment); NOT the catalog.

The mode is mechanically a **per-turn parameter**, conceptually **per-conversation / per-deployment** —
exactly like `model` today. It rides the same three-layer path `model` already proves:

- **Core seam — `ProduceOptions.mode`.** `mode` joins `maxRounds`/`model`/`k` in `ProduceOptions`
  (`produce.ts:38-47`) — the per-turn tuning knobs, distinct from `ProduceDeps`' injected surfaces —
  and `produce()` passes it to `buildSystemPrompt` (`produce.ts:153`). The proxy reads `mode` from the
  POST body and validates enum membership (a bare `mode` string, defaulted if unknown — the trust
  boundary already parses `{input, provider, model}` at `dev-proxy-plugin.ts:108`), exactly as it passes
  the allowlist-validated `model` as the authoritative `opts.model` (`:140`). The live transport adds
  `mode` to the body it POSTs (`live-proxy-transport.ts:41-45`).
- **Demo surface — a dev-only switcher selector.** The switcher (`provider-switcher.ts`, LLD-C12) gains a
  `ui-select` for `mode`, persisted to localStorage and exposed on the selection ref — the SAME dogfooded
  pattern its provider/model selects already use (`provider-switcher.ts:92-134`). Dev-only, tree-shaken
  from the build like the rest of the overlay (SPEC-N2). This lets the live demo showcase all three use
  cases off one catalog.
- **Deployment default — the construction site.** A real consuming app sets its mode where it constructs
  the transport / calls `produce()`, and simply does not render the selector (a fixed-disposition
  deployment). The default is the balanced grammar unless overridden.

The catalog is deliberately NOT the home for the mode (see *Alternatives*): it is the sole COMPONENT
authority (SPEC-R6), and a UX disposition is not a component fact; overloading it also can't be changed
per-conversation and risks the drift-gated schema. A per-deployment default belongs at the construction
site, not baked into the component inventory.

## Open fork (Kim's call) — and the re-verify points it is *not*

Exactly **one** fork needs Kim's judgment; the rest are build-tuning settled empirically, and the default
is a principled back-compat choice, not a fork.

- **Fork (Kim's call) — is Structural a NAMED enum value, or documented separately? RESOLVED (Kim,
  2026-07-07): documented separately, per the planner's recommendation.** Two shapes were considered,
  both zero-code for Structural itself:
  - *(recommended)* A clean **two-disposition live-grammar axis** — `GenUiMode = 'default' | 'specific'
    | 'blue-sky'` — with Structural documented as "the recorded transport you already have" (Decision §3).
    Rationale: a transport choice and a grammar disposition are orthogonal axes at different layers;
    folding them into one enum puts a value on the prompt-composer's input that means "don't run the
    prompt composer," which the seam must special-case.
  - A **uniform three-value use-case enum** — `'specific' | 'structural' | 'blue-sky'` — matching Kim's
    own three-use-case mental model, where `structural` is a passthrough the page/proxy maps to the
    recorded path (skipping live generation). More legible in the demo's single selector; costs a
    special-case at the seam.
  It is Kim's call because it is product-framing taste with a real (small) consequence: whether the
  switcher offers two live options or three, and whether `GenUiMode` carries a `structural` member the
  live path must branch on. *Recommendation: the two-disposition axis + documented Structural* — keep the
  axis conceptually one thing (live disposition); revisit if Kim wants the uniform surface.
- **Build-time re-verify (NOT Kim's call) — the exact mode names, count, and per-mode wording.** Whether
  a fourth "neutral vs default" distinction is worth a name, and where exactly each mode's clarify
  threshold and negotiation appetite sit, is GRAMMAR prose tuned against real turns — the same
  vagueness-threshold re-verify ADR-0089 already booked, one axis wider. An observation, not a ruling.
- **Build-time re-verify (NOT Kim's call) — demo selector exposure.** The demo naturally shows the
  selector to showcase the modes; a real deployment hides it and fixes the mode at construction. The seam
  supports both; which the *demo* does is settled by building it.
- **Deferred refinement (NOT a fork) — a structured `noteKind` reasoning trace for blue-sky.** A
  distinct chat styling for "I'm reasoning through options" would reuse ADR-0089's already-parked
  additive `noteKind` field. v1 rides the plain note channel (sufficient — the note is shown verbatim);
  add `noteKind` only if plain narration tests poorly. Recorded as a refinement, not built now.

## Consequences

- **A deployment can finally pick its disposition.** A curated chatbot gets fast directive mapping; an
  open-ended playground gets an exploratory reasoner; both off one catalog and one channel. The "config
  or flag mechanism that enables the more open-ended use cases" Kim asked for.
- **Zero new wire/transport/protocol surface, and the drift gate is untouched.** The axis is a
  grammar-half composition + one `ProduceOptions` field flowing the proven `model` path. `AgentTransport`,
  the meta-line shape, `produce()`'s peel/validate/stream, and the reducer are all exactly as ADR-0088/0089
  leave them; `prompt-drift.test.ts` asserts only the catalog-derived inventory, which does not change.
  This cheapness is a direct consequence of ADR-0088/0089 having built the channel + grammar seam first.
- **Structural cost is a doc, not a build.** The most "structural" of Kim's three use cases turned out to
  be the cheapest — it is already shipped; the only debt is a doc/example and the standing validity gate
  on that example.
- **Hard dependency on ADR-0089 (and thus ADR-0088).** The axis SCALES 0089's clarify/negotiate grammar;
  without those behaviors built there is nothing to scale. Sequence 0088 → 0089 → 0090.
- **Honest costs / new behavior to enumerate in the LLD:**
  - **Default byte-identity is load-bearing and must be tested.** The mode composition, at the default
    mode, MUST emit the exact pre-mode ADR-0089 grammar string — otherwise the "zero regression" claim is
    hollow and the drift-adjacent `system-prompt-grammar.test.ts` assertions (SPEC-R6 AC2) silently shift.
    A byte-identity assertion for the default is the mitigation; it is a real test-surface cost of moving
    a `const` string into a composer.
  - **Quality per mode is not gate-covered.** Whether `specific` feels appropriately directive and
    `blue-sky` appropriately exploratory is live-model prose — ungated at runtime (ADR-0070: quality is
    authoring/curation-time). A mis-tuned mode is a prompt regression caught by observation, not a test.
  - **The mode is trusted input at a security-adjacent boundary.** The proxy must validate `mode`
    membership and default an unknown value — a crafted `mode` must never reach the prompt raw. This is
    cheaper than the `{provider,model}` allowlist (an enum, not a registry) but must not be skipped; the
    honesty floor (§2) means even a mis-set mode can never breach the catalog allowlist.
  - **Two dispositions can drift from the default.** Once the grammar is three variants, a future
    ADR-0089-style grammar edit must update all three (and re-assert the default's byte-identity), or the
    variants rot apart. The composer's shared spine (§1/§2) limits the blast radius to the scaled block.
- **Stale → re-verify on the build gate:** the default-mode byte-identity assertion (if a later ADR edits
  the grammar) · the per-mode wording (tune against real turns) · whether `noteKind` styling is needed
  (try plain first) · the Structural example's validity gate (re-validate on a catalog version bump, like
  any committed transcript) · the inline-vs-corpus split for blue-sky's calibration examples (measure
  per-turn token cost against blue-sky quality; the same class of build-tuning as the per-mode wording) ·
  whether a curated catalog-scoped pattern-knowledge corpus (a `/ui-patterns` analog) is later warranted
  (a flagged follow-up, not this decision — see *Calibration examples & the deferred corpus question*
  below).

### Calibration examples & the deferred corpus question (blue-sky's dual-direction enrichment)

Five worked *need → parts → catalog mapping → what falls to the wall* examples, grounded in the real
post-ADR-0087 catalog (Text, Button, TextField, Field, FormProvider, Checkbox, Switch, Select/Option,
Row, Column, Card family, Tabs, Modal, Icon, Menu, Popover, Tooltip, RadioGroup, Slider, SliderMulti,
Calendar, ComboBox, List, Grid — no table/chart/map/carousel/drawer primitive). ~2-3 (marked ★) ship
inline in the blue-sky grammar; the rest seed as retrievable corpus exemplars, to bound token cost — the
inline/corpus split itself is a build-time re-verify, not fixed here.

1. **★ A card-game component sheet** (Kim's own example). Parts: hand, discard/deck pile, score readout,
   action bar. Map: hand = `Row(gap)` of `Card`s · pile = a `Card` with a `Text` count · score = a `Grid`
   of `Text` (name→value) · action bar = a `Row` of `Button`s. **Wall:** drag-to-reorder, card-flip
   animation, and playing-card face art are not hosted — name them in the note, render static `Card`s as
   the approximation. *(the unhosted-part → ADR-0089 catalog-wall path)*
2. **★ A settings screen.** Parts: grouped preference rows + save. Map: `Card` › `CardContent` › `List`
   of `Field` (label+description) each wrapping a `Switch`/`Select`/`TextField`; `CardFooter` › `Button`.
   **Wall:** none — fully hosted. *(a clean load-bearing case; `Field`+`List` is the settings idiom)*
3. **★ A dashboard / summary.** Parts: 3-7 KPI stats. Map: `Grid(min)` of `Card`s, each a `Column` of
   `Text variant="caption"` (label) + `Text variant="h2"` (value). **Wall:** real charts/sparklines are
   not hosted — offer the numeric grid, name the missing chart. *(a second, distinct wall case)*
4. **A sign-up / login form.** Parts: titled section, labelled validated inputs, submit. Map:
   `FormProvider` › `Card` › `CardHeader`(`Text h3`) + `CardContent`(`Column` of `Field` › `TextField`
   with `required`, `type=email`/`password`, wired to the catalog's `required`/`email` functions) +
   `CardFooter`(`Button` action=submit). **Wall:** none. *(shows the validation-function wiring)*
5. **A master-detail / list-and-inspector split.** Parts: scannable list + selection-bound detail. Map:
   `Row` split — left `List` of selectable rows; right `Card` detail bound via an `updateDataModel` path.
   **Wall:** true two-way selection sync is a live `action`→`updateComponents` round-trip (hosted by
   ADR-0088, but a round-trip, not instant) — worth naming. *(shows binding + the round-trip)*

**Deferred, not decided here — a curated, catalog-scoped pattern-knowledge corpus** (a `/ui-patterns`
analog trimmed to A2UI's ~32 primitives, roughly 10-15 entries in the anatomy → when-it-fits → failure
shape). This is honestly-sized curation work — comparable in kind to this session's own
`a2ui-training-corpus` curation effort, not a "small v1" item, and it is deliberately NOT bundled into
this ADR. If later commissioned, the vehicle is itself a sub-question: mint entries as retrievable corpus
exemplars (reuses the shipped `retrieve()` path, zero new runtime mechanism — recommended) vs. an
authoring-time knowledge pack consumed by whoever curates the corpus (never by the live model itself, per
§1's Skill-path finding). Recommendation: ship this ADR's enrichment now (zero new mechanism, delivers
the capability); decide the corpus on its own merits after observing blue-sky against real turns — the
calibration examples + existing retrieval likely cover the common cases, and the corpus is worth its real
cost only if observation shows recurring catalog-composition misses.

## Acceptance

*(Predicates for the eventual build, if ratified — not run here; this is a design record. All assume
ADR-0088 + ADR-0089 are built.)*

- `buildSystemPrompt(catalog, [], /* default */)` returns a string byte-identical to the pre-mode
  ADR-0089 grammar; `buildSystemPrompt(catalog, [], 'specific')` dials the negotiated-approximation block
  down and `('…','blue-sky')` lowers the clarify threshold; ALL three contain the never-invent-a-type
  honesty floor and the `## Available components` inventory carries none of the mode prose — a standing
  `system-prompt-grammar.test.ts`, `npm test` green, no live model. `prompt-drift.test.ts` stays green.
- A stub-provider `produce()` run with `opts.mode='specific'` feeds the specific grammar to the provider,
  and absent `mode` reproduces the default — deterministic unit, `npm test` green.
- A read confirms `mode` rides only `ProduceOptions` + the proxy body + the live-transport body (the
  `model` path); `AgentTransport.turn`, the meta-line shape, `produce()`'s validate/stream, and the
  reducer are unchanged in the diff. An unknown `mode` in the body is defaulted/rejected, never forwarded raw.
- A read confirms no path renders or admits an uncatalogued type in ANY mode (the SPEC-R9 allowlist is
  mode-invariant; the honesty floor is in the shared spine).
- The Structural doc's worked transcript passes `validateA2ui` under a standing test and renders through
  `createRecordedTransport` with no live model (`recorded-transport.ts:25-36`) — no runtime code change;
  `round-trip.test.ts` stays green.

## Alternatives considered

- **Amend ADR-0089 in place** (append a `## Amendment`) rather than a new ADR. Rejected by the log's own
  amendment-vs-supersession test (`README.md` §"Amendment vs supersession"): an amendment lands "an
  extension the decision **already anticipated**." ADR-0089 shipped the ASK grammar as UNCONDITIONAL and
  its own re-verify points are all prompt-tuning (vagueness threshold, `noteKind`, backbone seed) — it did
  not anticipate a config/mode axis. Its Decision *stands* (the default mode reproduces it), and this is a
  *separate, new* decision built on it → the log's **Extension** case (new ADR, two-way `Extends`/`Extended
  by` link), exactly as ADR-0089 itself extends ADR-0088 one rung down.
- **Supersede ADR-0089.** Rejected: nothing in 0089 is reversed. The default mode is 0089's grammar
  verbatim; the modes scale it. A decision that is preserved as the default is extended, not superseded.
- **Put the mode on the catalog / its registration** (a `genUiMode` hint in `catalog.json`). Rejected:
  the catalog is the sole COMPONENT authority (SPEC-R6); a UX disposition is not a component fact, cannot
  be varied per-conversation (a demo showing both modes off one catalog could not), and risks the
  drift-gated schema. A per-deployment default belongs at the construction site (Decision §4).
- **Build-time / server-side config only** (a `providers.json`-style registry, no user surface).
  Rejected: `providers.json` is the provider/model/key registry — mode is orthogonal, and a build-only
  value cannot let the demo showcase the modes. The core seam + a dev selector is barely more code and far
  more useful.
- **Make Structural a full live mode** (a prompt disposition that generates "structural" output live).
  Rejected: it misreads Kim's intent — Structural is "render pre-generated JSON," i.e. no live model at
  render time, which the recorded transport already is. Building a live "structural mode" would be new
  mechanism for a capability that already ships as a transport.
- **A mode that can widen the catalog allowlist** (e.g. `blue-sky` may emit new types). Rejected on
  ADR-0089's own two grounds: it is a security decision misfiled as UX, and it is mechanically
  self-defeating (an uncatalogued type fails the shared validator and never streams). The honesty floor is
  mode-invariant (Decision §2).

## Out of scope (this ADR)

- **Any widening of the render-time security allowlist (SPEC-R9), in any mode.** A firm NO — the honesty
  floor is mode-invariant; "blue-sky" scales exploration, never the allowlist. Catalog membership is a
  catalog-design decision (`a2ui-catalog-design`/ADR-0087 territory), never a mode knob.
- **Multi-turn planning / agent-held plan state.** `blue-sky` welcomes several clarifying ROUNDS (each a
  note-only turn on the existing channel), but there is still no dialog tree, no plan object — as in
  ADR-0088/0089.
- **A new wire/transport/protocol surface.** None; the mode rides `ProduceOptions` + the request body.
  A `noteKind`-styled reasoning trace is a deferred refinement (ADR-0089's parked field), not part of this
  decision.
- **A rubric for per-mode quality.** Runtime quality stays ungated (ADR-0070).
- **Building out the Structural pattern's consuming apps.** This ADR names Structural + ships one worked
  example; real structural deployments are downstream PRD-G2 consumers, unbounded (PRD out-of-scope).
- **Any new LLM capability** beyond "compose the grammar per a mode" — no tool use, no second model call
  (prompt engineering only).

## Build-sequencing note (indicative size, if ratified — a full decomposition is a follow-up)

**Depends on ADR-0089 (and ADR-0088) being built first.** Given that, ADR-0090 is small — roughly,
smallest→largest (the finalized decomposition manifest, both planes coverage-clean at PLAN mode, backs
this):

1. **`GenUiMode` type + default** (new, tiny): a union + default constant in `tools/agent/` (near
   `system-prompt.ts`). Types-only; `npm run check`.
2. **Mode-composed GRAMMAR** (`system-prompt.ts`, LLD-C4): split the `const` GRAMMAR into an invariant
   spine (note line + honesty floor + output rules) and a mode-scaled block (clarify + negotiate);
   `buildSystemPrompt(catalog, exemplars, mode)` composes them. Assert default byte-identity + per-mode
   variants in a new `system-prompt-grammar.test.ts`; `prompt-drift.test.ts` unaffected (asserted). **The
   bulk of the change.**
3. **`produce()` threads `opts.mode`** (`produce.ts`, LLD-C3): pass `opts.mode` into `buildSystemPrompt`;
   absent ⇒ default. Extends `produce-loop.test.ts`.
4. **Proxy reads + validates `mode`** (`dev-proxy-plugin.ts`, LLD-C6): parse `mode` from the body, default
   an unknown value, pass as `opts.mode` — the `model` precedent.
5. **Live transport sends `mode`** (`live-proxy-transport.ts`, LLD-C7): add `mode` to the POST body from
   the selection ref.
6. **Demo mode selector** (`provider-switcher.ts`, LLD-C12, `check:site`): a `ui-select` for mode,
   persisted, dev-only — the ADR-0089 switcher precedent. New jsdom test.
7. **Structural doc + worked transcript example** (LLD-C2): a doc naming Structural as the recorded-transport
   pattern + a second committed transcript, gated by `validateA2ui` (the `examples.test.ts`/`round-trip`
   precedent). Zero runtime code.
8. **SPEC/LLD repairs** (SPEC-R6/R8/R12/R2, LLD-C2/C3/C4/C6/C7/C12) landed with the build, per this ADR's
   `Repairs:`; `trace_check.py` green.

The heaviest slice is (2) the grammar composition; the plumbing (3–5) mirrors the `model` path and (7) is
doc-only. This is deliberately a cheap axis — the channel (ADR-0088) and the ASK grammar (ADR-0089) were
already paid for.
