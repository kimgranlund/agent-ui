## Context

The ADR-0079 gallery dogfood put four **bare** `ui-select`s in the toolbar (the Scheme / Scale / Density / Theme axes), replacing native `<select aria-label="…">`. Each select announces only its current **value** ("light" / "ui-md" / …) with **no axis name** — a real accessibility regression from the native control it replaced.

Root cause: `ui-select` has **no label pass-through**. The host carries `aria: role: none` (a logical select wrapper — `select.md:99`), so an `aria-label` set on the host is **inert**: it never reaches the interactive trigger. The trigger is a `<button data-part="trigger">` whose accessible name comes from **name-from-content** = the visible `[data-part="label"]` span, i.e. the value text (`select.ts:173–189,281`). There is no seam that gives the trigger an axis name distinct from its value.

`ui-select` **extends `UIFormElement`**, so it already inherits the ADR-0051 field-labelling seam — but that seam has two holes here:

- **(a) the base default no-ops for a role-less host.** ADR-0051 cl.2's base `applyFieldLabelling` forwards via `internals` ARIA-element reflection **only when `internals.role` is set**; the select host is deliberately role-less, so the default forwards nothing. This is exactly ADR-0051's own Consequences note — *"combo-box and select are flagged follow-ups … until wired, a select inside a field keeps only its own trigger-text name, and the bridge cannot help select (it has no `label` prop)."*
- **(b) the seam covers only FIELDED usage.** ADR-0051 is about `ui-field` association; it says nothing about a **bare** select (no field) — which is precisely the gallery's case. Bare labelling is what text-field's separate `label` prop covers (ADR-0014).

The button pattern also **breaks text-field's simple stand-in**. text-field's bare seam is `label` prop → the editor's `aria-label` (ADR-0014). That works because a **textbox has a distinct accessible value** (its text content) that survives an `aria-label` name. A `<button>` has **no distinct value AX property** — its content *is* its name — so an `aria-label` on the trigger would **overwrite** the value in the accessible name (accessible-name precedence: `aria-labelledby` > `aria-label` > content). Announcing "Scheme" while silently dropping "light" is a different, equal defect. The seam must yield an accessible name that carries **both** the axis name and the value.

`ui-combo-box` shares the overlay/trigger family and the same missing seam, but **not the same ARIA shape**: its interactive part is a `<div data-part="editor" role="combobox" contenteditable>` (`combo-box.ts:270–306`) whose accessible **value** is its text content (like a textbox). A bare combo-box editor today has **no accessible name at all** (a `combobox` does not take its name from content — content is the value), a strictly worse gap than select's value-only name. So combo-box needs the same seam — but its correct mechanism is text-field's (editor `aria-label`), **not** the button concatenation select needs.

## Decision

We will add a labelling seam to `ui-select` and `ui-combo-box`, in **two parallel clauses that share a `label` prop API but differ in ARIA wiring** — because a trigger **button** has no distinct value and a combobox **editor** does. Both **consume ADR-0051's `applyFieldLabelling` override** for the fielded path; both add a `label` prop for the bare path — the same dual structure text-field ships. The owning descriptor docs (`select.md`, `combo-box.md`) are repaired (the `labelSource` clause + the `attributes[]` block); `select.ts` / `combo-box.ts` gain the wires (build-time, gated on this ADR). Five clauses:

1. **A `label` prop on each control (the bare-usage name source).** `label: prop.string('')`, `reflect:false` — an accessibility hint, not a styling hook (the text-field precedent). It is **the** API; a host `aria-label` attribute stays inert (role-less host) and is not the seam. `''` = no label → the trigger keeps its content-only accessible name (back-compat).

2. **`ui-select` — the trigger's accessible name is `aria-labelledby` CONCATENATION of the label + the value span (never `aria-label`).** The value span `[data-part="label"]` is assigned a stable id once. A scope-owned effect maintains the trigger's `aria-labelledby`:
   - `label` set, not fielded → the control creates/updates a **visually-hidden `[data-part="aria-label"]` span** holding the label text, and sets `trigger.aria-labelledby = "<aria-label-span-id> <value-span-id>"` → accessible name = "Scheme light" (axis + value).
   - `label` empty, not fielded → `aria-labelledby` removed → content-only accessible name (today's behavior; zero drift).
   The value stays **in** the accessible name because the value span is the second reference — and it recomputes **live** as selection changes (`aria-labelledby` re-reads referenced text). An `aria-label` on the trigger is explicitly rejected here (it would erase the value — see Alternatives).

3. **`ui-combo-box` — the editor takes `aria-label` directly (text-field parity).** The editor is `role="combobox"` with a **distinct** value (its text), so `label` prop → `editor.setAttribute('aria-label', this.label)` gives name = label, value = editor text — both preserved, exactly ADR-0014. **No** concatenation node is needed. Guarded by `this.fieldLabelling === null` so it yields to the fielded path (accname precedence), mirroring `text-field.ts:425`.

4. **Fielded path — both override `applyFieldLabelling` (the ADR-0051 part-role reference); select MERGES.** Inside a `ui-field`:
   - `ui-combo-box` mirrors text-field exactly (`text-field.ts:662–673`): id-reference the editor's `aria-labelledby` ← `refs.label`, `aria-describedby` ← `[refs.description, refs.error]`; the bare `aria-label` yields.
   - `ui-select` **MERGES, never clobbers** (the ADR-0051 calendar-merge note, cl. Consequences): `trigger.aria-labelledby = "<refs.label.id> <value-span-id>"` (field label + value — the **same shape** as the bare path, only the name source swaps from the hidden span to the field's visible label element); `aria-describedby` ← `[refs.description, refs.error]`. On dissociation (`refs === null`) it reverts to the clause-2 bare state.
   The trigger **has no internal validity-message node** (unlike the text-field / combo-box editors), so ADR-0051's single-announce "yield" (cl.4) is a **no-op** for select — its `aria-describedby` is purely the field's error/description.

5. **Scope: combo-box is IN — this ADR closes the ADR-0051 "select/combo-box labelling wires" follow-up entirely.** Only `ui-select` is **gallery-blocking** (the four toolbar axes). `ui-combo-box` is not in the gallery, but its bare editor has **no accessible name today** — a real, strictly-worse gap — so it is fixed in the **same record**, and co-locating both is what lets the record own the button-vs-editor ARIA asymmetry that is this decision's substance. `ui-menu` / `ui-popover` / `ui-tooltip` are **OUT**: menu items self-name from content, and popover/tooltip are not form controls with a value/label duality.

## Consequences

- **The gallery fix (downstream, not this ADR's build):** the toolbar's four `ui-select`s set `label="Scheme|Scale|Density|Theme"`; the honest "no label pass-through" gap comment in `site/lib/component-gallery.*` is removed. `select.ts` / `combo-box.ts` gain the prop + effect + `applyFieldLabelling` override; the descriptor-doc repairs (each `labelSource` + the `attributes[]` / `parts[]` rows) are detailed in the two bullets below. All build-time, gated on this ADR.
- **`select.md`'s `labelSource` clause CHANGES (from accurate → new contract).** Today's clause (`select.md:101`) is *accurate* — *"The trigger's visible `[data-part=label]` text provides the accessible name via its text content"*. It becomes the concatenation contract: bare (labelled) = `label` + value via `aria-labelledby`; fielded = field label + value; unlabelled = content-only (back-compat). The trip-wire that mirrors `attributes[]` ↔ `static props` (family-coherence, ADR-0081) requires the new `label` row — **and** `select.md` gains a `parts[]` row for the NEW visually-hidden `[data-part=aria-label]` span (clause 2), which the parts/contract trip-wire will otherwise flag as an undocumented part.
- **`combo-box.md`'s `labelSource` clause is a DRIFT CORRECTION (from phantom → real).** Unlike select, today's clause (`combo-box.md:99`) is *drifted* — it claims *"aria-label on the editor part (set by the author via the `label` slot or an explicit aria-label), or an associated `<label for=…>`"*, a labelling path the code never had (`combo-box.ts` has no `label` prop, no slot, no `aria-label` wiring — every "label" there is `#labelForValue`, the display value; a light-DOM `<label for>` cannot reach the internal editor part). That phantom contradicts this ADR's Context ("no accessible name at all today"). The repair REPLACES it with the real mechanism: bare = `label` prop → editor `aria-label` (ADR-0014 parity); fielded = the field's label via `applyFieldLabelling`. This is a correction of a lying doc, not a from-accurate addition.
- **The asymmetry becomes a documented family invariant:** a **trigger-button** control concatenates (no distinct value); an **editor/combobox** control uses `aria-label` (distinct value). A future overlay control with a button trigger inherits clause 2; one with an editor inherits clause 3.
- **jsdom cannot verify an accessible name** (no accname computation) — the read-back (select "Scheme light", combo-box "Scheme" + editor value) is provable only in the **browser gate, both engines** (the ADR-0051 precedent). jsdom probes cover the **attribute wiring**: the id list on `aria-labelledby`, the `aria-label`-absence guard on the select trigger, the yield, and the merge-not-clobber.
- **Zero drift for unlabelled/unfielded usage:** no `label`, no field → no `aria-labelledby` / `aria-label` set → today's content-only name. The full select/combo-box suites stay green.
- **`aria-labelledby` self-recompute is a feature:** the trigger's accessible name tracks the value **live** (AT re-announces on pick) — an improvement over a static `aria-label`, and the reason concatenation beats a value-echo string the control would have to keep in sync.
- **Stale → re-verify:** `controls/select/{select.ts,select.md}` · `controls/combo-box/{combo-box.ts,combo-box.md}` · the gallery toolbar + its gap comment · **ADR-0051** (gains the `Extended by ADR-0085` back-link in its header + its Consequences select/combo-box follow-up line, now closed — both on ratification) · any goals/coherence note referencing "select/combo-box labelling wires".

## Acceptance

- **Bare select accessible name (browser, both engines):** `<ui-select label="Scheme">` with "light" selected exposes accessible name "Scheme light" (axis + value); changing the selection to "dark" updates it to "Scheme dark" **live**.
- **Bare select back-compat:** `<ui-select>` with no `label` keeps a content-only accessible name (the value text); no `aria-labelledby` / `aria-label` is set on the trigger.
- **No `aria-label` on the trigger (the value-erasure guard):** the trigger never carries an `aria-label` attribute even when labelled — an attribute probe asserts its absence.
- **Bare combo-box (browser, both engines):** `<ui-combo-box label="City">` exposes accessible name "City" on the editor, with the editor's text as the (distinct) value; no `label` → the editor keeps its ADR-0014-style empty/placeholder behavior unchanged.
- **Fielded select (browser, both engines):** inside `ui-field label="Scheme"`, the trigger's accessible name is "Scheme" + value via `aria-labelledby` (**merge, not clobber**); the field's error node is the trigger's sole `aria-describedby` member.
- **Fielded combo-box:** the field's label is the editor's accessible name (`aria-labelledby` id path); the bare `aria-label` yields; single-announce error via the field's description node.
- **Merge-not-clobber (jsdom):** `setFieldLabelling` on a select preserves the value-span id in `aria-labelledby` (asserts two ids, field-label first, value second); dissociation reverts to the bare state.
- **Zero drift:** existing select/combo-box suites stay green with no `label` and no `ui-field`.

## Alternatives considered

- **`aria-label` on the trigger (text-field's bare mechanism) for select** — rejected: a `<button>` has **no distinct value AX property**, so `aria-label` overwrites the value in the accessible name (accname precedence). Announcing "Scheme" while dropping "light" is an equal, opposite defect. It works for **combo-box** (clause 3) precisely because its editor **has** a distinct value; it does not for the button.
- **A bespoke `label`-only prop with no ADR-0051 reuse** — rejected: it would fork the fielded path off the fleet's one association seam, re-introducing the double-announce problem ADR-0051 solved. The design reuses `applyFieldLabelling` (fielded) and adds `label` **only** for the bare path — the same dual structure text-field ships.
- **`aria-labelledby` self-referencing the trigger's own id** (instead of the value span) — rejected as brittle: self-reference is under-supported in some AT and pulls in the whole trigger subtree (incl. the caret, though it is `aria-hidden`); referencing the explicit `[data-part="label"]` value span is precise and excludes the caret cleanly.
- **A value echo into the label string** (the control keeps `aria-label = "Scheme " + value` in sync) — rejected: it duplicates the value into a second source the control must re-sync on every selection change; `aria-labelledby` referencing the live value span gets the same result for free and cannot drift.
- **Punt combo-box to a named follow-up** — rejected: it perpetuates the exact ADR-0051 follow-up this record exists to close, and it orphans the button-vs-editor asymmetry (the decision's core content); combo-box's bare editor has no accessible name today — a real gap. Only the **wiring** differs (clause 3 vs clause 2); one record owns both.
- **Re-architecting the trigger to `role="combobox"` on the button** (to gain a distinct value) — rejected as out of scope and semantically wrong: the button + `aria-haspopup="listbox"` disclosure pattern is valid and shipped (ADR-0043); `role="combobox"` on a non-editable button invites AT to expect editability. The label seam does not require it.
