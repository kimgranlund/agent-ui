## Context

Task #9 (the renderer LLD-C10 function evaluator, `functions.ts`) was grounded against the repo at HEAD `8936553` and against the **A2UI v1.0** function / data-binding / expression facts fetched from a2ui.org/specification/v1.0 and supplied by team-lead (Constraint **C1** — conform, cited-not-redefined; the same external-grounding loop that produced ADR-0024). Seven discovered-reality facts the repo did **not** yet model:

1. **A binding value has a THIRD kind.** Beyond a literal and a `{path}` data-binding, a binding value can be a **function call**: `{ "call": "<name>", "args": { …named… } }` (optionally with a sibling `"message"` for a check). `protocol.ts:29` and runtime SPEC §5.1 model only `T | { path }`.
2. **Call args are a NAMED OBJECT, not positional.** e.g. `{ "call":"required", "args":{ "value":{"path":"/formData/zip"} } }`. The repo's `FunctionDef.args: JsonSchema[]` (`catalog.ts:35`) is a **positional array** — non-conformant for v1.0.
3. **Two function namespaces.** **SYSTEM** functions are `@`-prefixed (reserved at catalog load — `naming.ts:5`, so they cannot collide with a catalog name); **CATALOG** functions are keyed by name in the bound catalog's `functions` registry (`catalog.ts:14`, `Catalog.functions`).
4. **`@index` is the ONLY system function in v1.0, and it is INNERMOST-ONLY.** It "MUST ONLY be available when evaluating template items within a list rendering context (Collection Scope); outside iteration the client MUST treat it as an error." Its `offset` arg is a **numeric addend** to the innermost iteration index (e.g. 1-based display), **not** outer/ancestor-scope addressing.
5. **Args recurse.** A call's args may each be a literal, a `{path}` binding, **or a nested `{call}`** — resolved recursively.
6. **String composition is `${…}` DynamicString interpolation, not a `formatString` function.** A DynamicString value embeds `${/path}` / `${fn(args)}` (with `\${` escape) — e.g. `"Hello, ${/user/firstName}! Welcome to ${/appName}"`. This is **richer** than, and supersedes, the repo's planned `{0}`-positional `formatString(tmpl, ...args)` catalog function (catalog LLD-C7 / catalog SPEC-R5).
7. **`checks` is a component-level array.** `checks: [{ call, args, message }]` producing a `{ code: VALIDATION_FAILED, surfaceId, path, message }` on failure; **where** a field's check message renders is **underspecified** on the v1.0 page.

The repo today **mis-handles** the unmodeled `{call}` value at three existing gates, which is the integration surface this ADR repairs:
- `widget.ts:31/81` — `isBinding` tests only `.path`, so a `{call,args}` value falls to the `else` branch and is `applyProp`'d **raw** onto the DOM prop (treated as a literal object).
- `conformance.ts:36-37` — `matchesType` accepts `{path}` on a `bindable` prop, else falls to `matchesSchemaType`; a `{call}` object on a string-typed prop → **type mismatch → a false `CATALOG` error**.
- `validate.ts:231` — `isBinding` (only `.path`) ignores `{call}`; harmless for the POINTER scan, but conformance composing into `validate.ts:156` inherits the false-CATALOG above.

## Decision

We add the **function-call binding value kind** to A2UI rendering and evaluate it in a new **`renderer/functions.ts`** (LLD-C10), the read-side twin of the `{path}` resolver (`binding.ts:131 resolve`):

1. **One value dispatcher.** A bound prop value is `literal | {path} | {call}`. A single `resolveValue(value, surface, itemScope?)` dispatches: a literal returns as-is; a `{path}` defers to `resolve` (`binding.ts`); a `{call}` defers to `evaluate` (`functions.ts`). The dispatcher runs **inside the existing scope-owned bound-prop effect** (`widget.ts:113`), so a call whose args read `{path}` re-resolves reactively and per-path waking (SPEC-N2) is preserved unchanged.
2. **The `{call}` split lives in `widget.ts`** — the same layer that already splits literal vs `{path}` (LLD-C7 / the LLD-C5 "as-built" note). `isBinding` widens to recognize `{call}` and routes it through the evaluator; conformance (`matchesType`) accepts `{call}` on a `bindable` prop as a **deferred-resolution** binding, symmetric with `{path}`. The shared validator stays pure/total — it adds no FUNCTION stage.
3. **`evaluate(call, surface, itemScope?)`** — dispatch on the name's namespace: `name[0] === '@'` → the **system table** (today: `@index`); else → `surface`'s bound `catalog.functions[name]` + the pure impl from `catalog/functions.ts`. **Args are a named object**, resolved **recursively** through `resolveValue` (literal | `{path}` | nested `{call}`).
4. **`@index` reads the innermost collection index — the deferred-from-list bit.** `@index` returns `itemScope.index + (args.offset ?? 0)`. The list already exposes the index via the single-frame `ItemScope` (`list.ts:103`, `types.ts:20`) threaded through `createWidget`/`tree.ts`/`list.ts` — `evaluate` reads it off the **same** signature `resolve` already takes. With **no** `itemScope` (outside a Collection Scope) `@index` is an **error** (FUNCTION) per v1.0.
5. **FUNCTION is a render-time emit, not a validator verdict.** An unknown/throwing function (a `@`-name not in the system table, a catalog name absent from `catalog.functions`, `@index` outside a list, or a throwing impl) emits `error{code:'FUNCTION'}` via the **same `emitError` sink** `widget.ts:69` uses for `CATALOG`, and yields `undefined` for the prop (render-time placeholder, consistent with an undefined `{path}` — SPEC-R4 AC2). For a `checks` entry it yields `{valid:false}` (checks-surfacing itself is a scoped follow-up, below).
6. **Named args — the catalog contract is corrected.** `FunctionDef.args` becomes **named** (`Record<string, JsonSchema>`) in `catalog.ts` + the catalog SPEC §5.1 + LLD-C7, matching the wire `args` object. Every default function signature is `(args: {…named…}) → value`.

**The ADR-0024 deferral is settled: single-frame `ItemScope`, no chain.** ADR-0024's subtree amendment booked "*an explicit scope-**chain** would be needed only if the LLD-C10 `@index` function ever had to address an **outer** loop's index … a **C10-era decision***." v1.0 `@index` is **innermost-only** (fact 4) and `offset` is a numeric addend, **not** outer-scope addressing — so the single-frame `{path,index}` is **sufficient and v1.0-faithful**; promoting `ItemScope` to a frame chain is **rejected** (YAGNI, and non-conformant — v1.0 has no outer-index addressing). Nested lists keep baking the outer indices into the inner item's `path` exactly as today.

### Scoped follow-ups (recorded here; not built by LLD-C10)

- **`${…}` DynamicString interpolation — a SEPARATE slice (my D5 call).** `${…}` is a **broad DynamicString-resolution feature** (any string-typed dynamic prop can embed `${/path}` / `${fn(args)}`), **not** merely `formatString`'s internals — the v1.0 example `"Hello, ${/user/firstName}!"` is a plain string with embedded refs, not a function call. It is a **string-template parser layered on `resolveValue`** (parse `${…}` segments → dispatch each through the evaluator) and is **orthogonal** to the `{call}` value kind. Therefore: **catalog SPEC-R5's `formatString` requirement is reconciled** — the default catalog ships `required`/`email`/`regex` (the pure validators), and string composition is delivered by the `${…}` follow-up (a project catalog MAY still register its own `formatString` function — the registry is open). This carve-out keeps LLD-C10 to the value-kind evaluator.
- **`checks`-surfacing — a dependent follow-up.** A component `checks: [{call,args,message}]` array runs each entry through **this** evaluator, but **surfacing** a failed check (the field error channel + the `VALIDATION_FAILED` error code, neither of which exists in the repo — the default catalog's TextField carries a boolean `required` prop, no `checks` array; `ErrorCode` has no `VALIDATION_FAILED`) is a separate task that **reuses** the evaluator. SPEC-R10 AC1 ("a `required` check surfaces its message on the widget") is met by that follow-up; LLD-C10 delivers the machinery it rides on.

## Build brief — for execution-lead (LLD-C10 + LLD-C7; multi-file, slice-ordered)

1. **`a2ui/catalog/functions.ts`** (NEW, LLD-C7) — pure, zero-dep, no DOM/signals: `required(args)→{valid,message?}`, `email`, `regex`. **Named args** (`Record<string,unknown>`) per the corrected `FunctionDef`. Tests: table-driven valid/invalid per fn.
2. **`a2ui/catalog/catalog.ts`** — `FunctionDef.args` positional `JsonSchema[]` → **named** `Record<string, JsonSchema>`; `validateFunctions` (`catalog.ts:161`) validates the named shape. Update `fixtures.ts:41` + the default `catalog.json` `functions` block to the named shape and the `required`/`email`/`regex` set.
3. **`a2ui/renderer/functions.ts`** (NEW, LLD-C10) — `evaluate(call, surface, itemScope?)`: `@index` (offset addend) → `itemScope.index + offset`, **error if no itemScope**; catalog name → `registry.get(surface.catalogId)?.catalog.functions[name]` lookup + the pure impl; unknown/throw → `emitError(FUNCTION)` + `undefined`. The `resolveValue(value, surface, itemScope)` dispatcher (literal | `{path}`→`resolve` | `{call}`→`evaluate`) lives here or in `binding.ts` (exec call); args resolve **recursively** through it.
4. **`a2ui/protocol.ts`** — `Binding<T> = T | { path: string } | FunctionCall`; `interface FunctionCall { call: string; args?: Record<string, Binding<unknown>>; message?: string }`. **`a2ui/renderer/widget.ts`** — the per-prop split recognizes `{call}` and routes it through `resolveValue` inside the bound-prop effect. **`a2ui/catalog/conformance.ts`** — `matchesType` accepts a `{call}` on a `bindable` prop (deferred resolution).
5. **Recursion** — args = literal | `{path}` | nested `{call}`; a `{path}` arg re-resolves reactively. Tests below.

**Tests** (jsdom + the kernel's `inspect`):
- **`@index` in a list (the deferred bit)** — a list-item bound to `{call:'@index'}` renders its 0-based index; `{call:'@index', args:{offset:1}}` renders index+1; **`@index` outside a list → FUNCTION**.
- **catalog function** — `required`/`email`/`regex` over a `{path}` arg returns the expected `{valid,…}`; reacts when the path changes.
- **unknown function** — a `@`-name not `@index`, and a catalog name absent from `functions`, each emit `FUNCTION` and render a placeholder (siblings unaffected — fault isolation, SPEC-N4).
- **recursion** — a call whose arg is a nested `{call}` evaluates; a `{path}` arg updates the result on `updateDataModel`.
- **conformance** — a `{call}` on a bindable prop no longer raises a false `CATALOG`.

Gate: `npm run check && npm test`.

## Consequences

- **The evaluator is a small twin of `resolve`** — one `resolveValue` dispatcher, a one-entry `@`-system table, a registry lookup. No new reactivity machinery: it runs in the bound-prop effect that already exists, so SPEC-N2 per-path waking holds for `{path}` args for free.
- **The binding contract is now three-armed** (`literal | {path} | {call}`) at every layer that reads a bound value (widget split, conformance, the SPEC §5.1 type, protocol.ts). The `{call}` arm is **opt-in by shape**, so every literal and `{path}` path is byte-for-byte unchanged.
- **The catalog `FunctionDef.args` change is breaking for the shape, not the callers** — `args` was positional in `catalog.ts`/`fixtures.ts`/`catalog.json` but had **no runtime consumer yet** (no `functions.ts` existed), so the positional→named correction touches only declarations + the loader's `validateFunctions`, not behavior.
- **`@index` is permanently innermost-only** — outer-index addressing is not a v1.0 concept; if a future protocol version adds it, that is a **new** ADR promoting `ItemScope` to a chain (ADR-0024's deferral is now *closed*, not merely deferred).
- **`formatString` leaves the catalog function set** — string composition is the DynamicString `${…}` follow-up. Catalog SPEC-R5's mandated set is corrected to `required`/`email`/`regex`; this is a **conformance correction**, not a feature cut (the capability moves to `${…}`, which is richer).
- **`checks` end-to-end is NOT delivered here** — SPEC-R10 AC1's *surfacing* needs the field-error channel + `VALIDATION_FAILED` (a new `ErrorCode`), tracked as a follow-up that reuses this evaluator. LLD-C10 delivers the evaluation, not the display.
- **Stale → re-verify (on ratify + build gate):** runtime SPEC §2/§5.1 (Binding arm) · catalog SPEC-R5 (function set + `${…}` note) · renderer LLD-C5 §5 + LLD-C10 §7 · catalog LLD-C7 §7 · the new `renderer/functions.ts` + `catalog/functions.ts` · `protocol.ts` Binding union · `catalog.ts` `FunctionDef` · `widget.ts` split · `conformance.ts` `matchesType` · `fixtures.ts` + `catalog.json` functions block.

## Alternatives considered

- **Promote `ItemScope` to a frame chain (a stack of `{path,index}`) so `@index` could address outer collection scopes** — **rejected**: v1.0 `@index` is innermost-only and `offset` is a numeric addend, not ancestor addressing (Constraint C1). A chain would be un-spec'd machinery threaded through five files for a capability the protocol does not grant. ADR-0024 correctly deferred this *exact* question to C10; the C10 answer is "no chain."
- **Keep `formatString` as a positional catalog function** (the repo's catalog LLD-C7 / SPEC-R5 plan) — **rejected**: v1.0's string-composition mechanism is `${…}` DynamicString interpolation, which is richer (embeds `${path}` and `${fn(args)}` mid-string) and is a string-resolution feature, not a function. Shipping a `{0}`-positional `formatString` would be a non-conformant parallel mechanism. (A project catalog may still register one — the registry is open.)
- **Fold `${…}` interpolation into this slice** (treat it as `formatString`'s internals) — **rejected**: `${…}` applies to **any** DynamicString prop, not just a `formatString` call, so it is a string-template parser orthogonal to the `{call}` evaluator. Bundling it would conflate a value-kind dispatcher with a string parser and bloat LLD-C10. Carved as a separate, evaluator-reusing follow-up (my D5 recommendation).
- **Evaluate function-call bindings in `validate.ts` / raise FUNCTION there** — **rejected**: `validateA2ui` is pure, total, and shared with corpus admission (SPEC-N6); a function's *resolution* (and `@index`, which needs the live render-time collection scope) is render-time, not a static verdict. FUNCTION rides the render-time `emitError` sink (like `CATALOG`), keeping the validator pipeline unchanged. The §9 error table already places FUNCTION at LLD-C10, not LLD-C11.
- **Make `{call}` valid on any prop (not just `bindable`)** — **rejected**: a function call is a *binding value*, so it occupies the same position as `{path}` and is gated by `bindable` symmetrically; a non-bindable prop accepting a computed value would diverge the conformance rule for no protocol reason.
