## Context

Task #13 is the follow-up ADR-0026 booked: "*`${‚Ä¶}` DynamicString interpolation ‚Äî a SEPARATE slice (my D5 call)‚Ä¶ a string-template parser layered on `resolveValue`, orthogonal to the `{call}` value kind.*" It was carved out so LLD-C10 stayed a value-kind evaluator; ADR-0026 also reconciled catalog SPEC-R5's old `formatString` requirement away in favor of this richer mechanism. This ADR delivers the carve-out. It is grounded against the **A2UI v1.0** DynamicString facts fetched from a2ui.org/specification/v1.0-a2ui/ and supplied by team-lead (Constraint **C1** ‚Äî conform, cited-not-redefined; same external-grounding loop as ADR-0024/0026).

The discovered-reality facts the repo does **not** yet model:

1. **A literal string is no longer always opaque.** `DynamicString` is a schema union (literal | `{path}` | `FunctionCall`), but at runtime *"any string containing `${‚Ä¶}` triggers interpolation."* So the literal-string arm of `resolveValue` (`functions.ts:72` ‚Äî today `return value`) must detect an embedded **unescaped** `${` and interpolate, rather than passing the raw template to `applyProp`. This applies to **any** bindable string prop (Text's `text`, a `placeholder`, a `label`), not a single function ‚Äî *"any property that can be bound to data is defined as a Dynamic\* type."*
2. **Syntax = `${‚Ä¶}` segments.** A template is a string interleaving literal runs with `${expr}` expressions. Inside an expression v1.0 admits **two** forms: (a) **JSON-Pointer paths** ‚Äî absolute `${/user/firstName}` or relative `${firstName}` in a collection scope; (b) **function calls in an EXPRESSION grammar** identified by parentheses ‚Äî `${now()}`, `${formatDate(value:${/currentDate}, format:'yyyy-MM-dd')}`, with nesting (`${‚Ä¶}` inside an arg). The expression-grammar function form is a **different surface** from the JSON `{call, args:{named}}` object ADR-0026 built.
3. **Escaping.** *"To include a literal `${` in a string, it must be escaped as `\${`."*
4. **Coercion (spec-exact table).** Numbers/Booleans ‚Üí standard string representation; **Null/Undefined ‚Üí empty string `""`**; Objects/Arrays ‚Üí JSON-stringified.
5. **Reactivity.** A template *"re-resolves when an embedded path's data changes"* (the v1.0 "Text label updates as the TextField types" cross-binding example).
6. **Relative paths.** `${name}` relative in a collection scope resolves via the same scope rewrite the list uses for `{path}` and `@index`.

The repo's integration surface for this is the **literal-string arm** of `resolveValue` (`functions.ts:72`). The reactive substrate (per-path memo, `resolve`, `scopedPointer`/`itemScope`) and the conformance gate are already in place from LLD-C5/C6 and ADR-0024/0026 ‚Äî this feature reuses them, it does not extend them.

## Decision

We add a **`${‚Ä¶}` DynamicString interpolator** as a small string-template parser layered on the existing `resolveValue` substrate, in a new **`renderer/interpolate.ts`**, hooked into the **literal-string arm** of `resolveValue` (`functions.ts`). `{path}` and `{call}` value-kind handling is byte-for-byte unchanged.

**Fork settled: (A) path-interpolation only this wave; defer the function-expression grammar.** The `${‚Ä¶}` expression form supports both JSON-Pointer paths and a function-call expression grammar (`name(arg:val)`). This wave ships **path interpolation** (absolute + relative-in-scope) + escaping + the full coercion table + reactivity, and **defers** the function-expression grammar (`name(arg:val)`, single-quoted string literals, nested `${‚Ä¶}` inside args) to its own follow-up (a new task: *function-call-in-string expression grammar*). Rationale: the function-expression form needs a real expression tokenizer (identifiers, parentheses, `arg:` named bindings, quoted literals, recursive `${‚Ä¶}`) that is distinct from and larger than the JSON `{call}` object ADR-0026 built, and its value is far lower than path interpolation ‚Äî path interpolation is exactly what the dynamic-list demo that triggered this whole track (a people-list rendering names/roles in `Text`) needs. The parser is designed **forward-compatible** (below) so the function arm lands as a classifier extension, not a teardown.

1. **Detection ‚Äî literal string with an unescaped `${`.** `isInterpolated(s)` returns true iff `s` contains a `${` not immediately preceded by a backslash. A literal string with **no** unescaped `${` falls through **byte-identical** (the `\${`-only and the ordinary cases are unchanged from today's `return value`).
2. **Scanner ‚Äî one escape-aware segment split.** `interpolate` scans the template into an ordered list of **literal runs** and **`${‚Ä¶}` expressions**: a `\${` emits a literal `${` (the backslash is consumed); an unescaped `${` opens an expression that closes at its matching `}` (brace-depth tracked so a future nested `${‚Ä¶}` is parse-stable). This single scanner is the load-bearing forward-compat seam ‚Äî it is unchanged when the function arm lands; only the per-expression **classifier** grows.
3. **Classifier ‚Äî path now, function-expr deferred.** Each `${expr}` body is classified: a body **without** a `(` is a **JSON-Pointer path** (this wave); a body **with** a `(` is the **function-expression form** (deferred). The classifier is the single point the function arm extends.
4. **Path resolution reuses LLD-C5.** A path expression resolves through `resolve({ path: expr }, surface, itemScope)` (`binding.ts`) ‚Äî the **same per-path memo** (`pathSignal`) the `{path}` value kind uses. An absolute path resolves from root; a **relative** path (no leading `/`) resolves within a list item's collection scope via `scopedPointer(expr, itemScope)` (ADR-0024) ‚Äî identical rewrite, read direction, to `{path}` and `@index`. No new resolution machinery.
5. **Coercion (spec-exact).** Each resolved value is coerced to a string per fact 4: `number`/`boolean` ‚Üí `String(v)`; `null`/`undefined` ‚Üí `""` (the empty-string sentinel, **not** the literal `"null"`/`"undefined"`); `object`/`array` ‚Üí `JSON.stringify(v)`; a resolved `string` ‚Üí itself. Segments are concatenated in source order.
6. **Reactivity is free.** `resolveValue` runs **inside the existing scope-owned bound-prop effect** (`widget.ts:125`). Because each `${/path}` reads through `resolve` (the per-path computed), the bound-prop effect depends on exactly the paths the template embeds ‚Äî a template with N paths re-resolves when **any** of the N changes, and an unrelated `updateDataModel` leaves it asleep (SPEC-N2 per-path waking, **reused unchanged**). This is the mechanism behind fact 5's "Text label updates as the TextField types."
7. **Conformance is a no-op ‚Äî confirmed.** An interpolation template is a `typeof value === 'string'` literal, so `conformance.ts matchesType` accepts it via the ordinary string-type leg (`matchesPrimitive` ‚Üí `'string'`). No `{path}`/`{call}` deferred-binding branch is involved, so **no conformance change is needed and no false `CATALOG` is raised** for a bindable string prop carrying a template. (Verified against `conformance.ts:47-49`.)

### Error / malformed handling ‚Äî render literally, no error code (design decision)

The v1.0 spec defines an escaping rule (`\${`) but does **not** define a render-time error code for a malformed interpolation template, and the only render-time code the repo has is `FUNCTION` (ADR-0026), which is for an unknown/throwing function-call *value kind* ‚Äî not for string parsing. We therefore **render the offending segment literally and emit no error** in two cases:

- **A deferred function-expression** (`${now()}`, body contains `(`): the unparsed `${‚Ä¶}` segment is emitted **verbatim** into the output string ‚Äî no error. Rationale: the function arm is simply not yet implemented; emitting an error code now would burn a code (INTERPOLATION/BINDING) that the spec does not define and that the function-arm follow-up would have to un-burn. When that follow-up lands, the **same** segment resolves instead of rendering literally ‚Äî a clean, additive upgrade with no behavior to retract.
- **A malformed/unterminated `${`** (no closing `}`): emitted **verbatim**, no error. The only spec escape mechanism is `\${`, so an unterminated `${` is most plausibly authoring text meant to display literally; tearing the string out of an otherwise-fine `Text` widget (or minting a new error code) is worse than showing it. This is consistent with SPEC-R4 AC2's render-time-placeholder discipline (an unresolvable binding is a placeholder, not an error) and SPEC-N4 fault isolation.

This is a defensible v1.0-faithful call, recorded as a design decision (the spec is silent on the malformed case). If a future protocol version defines an interpolation error code, that is a new ADR.

## Consequences

- **The literal-string arm of `resolveValue` is no longer a pure pass-through** ‚Äî it is `isInterpolated(s) ? interpolate(s, ‚Ä¶) : s`. Every non-`${` literal string is byte-identical to today (the guard short-circuits), so the `{path}`/`{call}` paths and all existing literal behavior are unchanged; only a string *containing* `${` changes.
- **A binding is no longer only an *object* ‚Äî a template *string* is bound too, so `widget.ts isBinding` must route it into the bound-prop effect** *(discovered in build)*. Clause 6's reactivity assumes the interpolator runs inside the scope-owned bound-prop effect (`widget.ts:125`), but what *routes* a value into that effect is `isBinding`: a value it rejects is `applyProp`'d **once** (static, render-once, never reactive). A template is a plain `string`, which the prior object-only `isBinding` (`{path} | FunctionCall`) rejected ‚Äî so **without** widening `isBinding` to also accept an interpolated string, a template would render once and **silently never re-resolve** (clause 6 would not hold). `isBinding` is therefore extended to `(v): v is {path} | FunctionCall | string` returning true when `v` is an object binding **or** `typeof v === 'string' && isInterpolated(v)`. Correctly scoped: a **non-template** literal string is **not** a binding and keeps the static `applyProp(once)` fast-path unchanged ‚Äî only a string carrying an unescaped `${‚Ä¶}` is now routed reactively. (The conformance gate is still untouched ‚Äî see clause 7; this widening is in the renderer's per-prop *routing* split, not the static validator.)
- **No new reactivity machinery, no new error code, no conformance change.** The interpolator is pure string-handling over the existing per-path resolver: SPEC-N2 waking, the `scopedPointer`/`itemScope` rewrite, and the bound-prop effect are all reused as-is. The Binding union (`protocol.ts`) is unchanged ‚Äî a DynamicString is still a `string` literal at the type level; "is it a template?" is a **runtime** property of the string, not a new wire shape.
- **The catalog SPEC-R5 `${‚Ä¶}` mention is now delivered, not promissory.** ADR-0026 reconciled `formatString` away "in favor of the `${‚Ä¶}` follow-up"; this ADR is that follow-up's path arm. String *composition* (the `formatString` use case ‚Äî `"Hello, ${/user/firstName}!"`) is now available; string *formatting* via functions in templates (`${formatDate(...)}`) waits on the deferred function-expression arm (a project catalog may still register a `{call}`-form `formatString` per ADR-0026).
- **The function-expression grammar is deferred, crisply.** A new task ‚Äî *A2UI `${fn(args)}` function-call-in-string expression grammar* ‚Äî owns the tokenizer (identifiers, parens, `arg:` named args, single-quoted string literals, nested `${‚Ä¶}` recursion) and routes a classified function-expression body to the `evaluate` arm of `functions.ts` (the same evaluator ADR-0026 built ‚Äî the grammar parses *into* a `FunctionCall`, then reuses `evaluate`). The scanner (clause 2) and classifier seam (clause 3) are built to absorb it without a rewrite.
- **Malformed/deferred templates render literally** ‚Äî a Text whose `text` is `${now()}` shows the literal `${now()}` until the function arm ships, and an unterminated `${` shows verbatim. This is intentional graceful degradation (no error spam, no torn UI), but it means a *typo'd path template* (`${/usr/name}` for a non-existent pointer) resolves to `""` (coercion of `undefined`), not an error ‚Äî consistent with SPEC-R4 AC2 for `{path}`.
- **Stale ‚Üí re-verify (on ratify + build gate):** runtime SPEC ¬ß2 (Binding ‚Äî literal-string-is-conditionally-a-template note) + SPEC-R10 (`${‚Ä¶}` clause normative) ¬∑ catalog SPEC-R5 (`${‚Ä¶}` delivered) ¬∑ renderer LLD-C5 ¬ß5 (literal arm interpolator) + LLD-C10 ¬ß7 (`${‚Ä¶}` status: path arm delivered, function arm deferred) ¬∑ the new `renderer/interpolate.ts` ¬∑ `functions.ts` (`resolveValue` literal arm) ¬∑ `conformance.ts` (confirm no change needed ‚Äî string literal leg).

## Alternatives considered

- **(B) Ship the full grammar now (path + function-expression + nesting)** ‚Äî rejected for this wave. The function-expression form needs a genuine expression tokenizer (parentheses, `arg:val` named args, single-quoted string literals, recursive `${‚Ä¶}`) that is a larger and distinct surface from both path interpolation and the JSON `{call}` object. Bundling it would delay the high-value, demo-driving path arm behind a parser whose value is far lower, and would conflate two parse problems. The scanner is built forward-compatible so (B) lands additively as a classifier extension ‚Äî deferring it costs nothing structural.
- **Mint an `INTERPOLATION` / `BINDING` error code for malformed or deferred templates** ‚Äî rejected. v1.0 defines no such code; the repo's only render-time code is `FUNCTION` (a value-kind concern). Rendering literally is graceful (SPEC-N4/R4 AC2), reversible (the deferred function arm upgrades the same segment with no retraction), and avoids burning a code the spec is silent on. If v1.0 later defines one, a new ADR adds it.
- **Reuse `FUNCTION` for a `${fn()}` segment encountered before the function arm exists** ‚Äî rejected. `FUNCTION` is for an unknown/throwing `{call}` value kind that the renderer *attempted to evaluate*; a `${fn()}` template the interpolator does **not yet parse** has not been evaluated and is not unknown ‚Äî it is unimplemented. Emitting `FUNCTION` would mislead the server into thinking its catalog function is missing. Literal render correctly signals "not interpreted."
- **Coerce `null`/`undefined` to `"null"`/`"undefined"`** (the na√Øve `String(v)`) ‚Äî rejected: violates the spec coercion table (fact 4), which mandates the empty string for both. A template like `"${/middleName} "` over absent data must render `" "`, not `"undefined "`.
- **Detect a template by parsing the schema `DynamicString` union (a new wire shape)** ‚Äî rejected. v1.0 says interpolation is triggered by the string *content* at runtime (`${‚Ä¶}` present), not by a distinct serialized type; a DynamicString literal is still a plain JSON string on the wire. Adding a wire shape would diverge from the spec and from the `protocol.ts` Binding union (which stays a three-arm `literal | {path} | {call}`).
- **Add a per-template memo (cache the parsed segment list per string)** ‚Äî rejected as premature. The scan is O(len) over a short prop string run once per `applyProp` inside an already-memoized effect that only re-runs when an embedded path changes; the per-*path* memo already prevents redundant pointer walks. A parse cache adds a map + key discipline for no measured cost. Revisit only if profiling shows the scan hot.
