## Context

Task #13 is the follow-up ADR-0026 booked: "*`${…}` DynamicString interpolation — a SEPARATE slice (my D5 call)… a string-template parser layered on `resolveValue`, orthogonal to the `{call}` value kind.*" It was carved out so LLD-C10 stayed a value-kind evaluator; ADR-0026 also reconciled catalog SPEC-R5's old `formatString` requirement away in favor of this richer mechanism. This ADR delivers the carve-out. It is grounded against the **A2UI v1.0** DynamicString facts fetched from a2ui.org/specification/v1.0-a2ui/ and supplied by team-lead (Constraint **C1** — conform, cited-not-redefined; same external-grounding loop as ADR-0024/0026).

The discovered-reality facts the repo does **not** yet model:

1. **A literal string is no longer always opaque.** `DynamicString` is a schema union (literal | `{path}` | `FunctionCall`), but at runtime *"any string containing `${…}` triggers interpolation."* So the literal-string arm of `resolveValue` (`functions.ts:72` — today `return value`) must detect an embedded **unescaped** `${` and interpolate, rather than passing the raw template to `applyProp`. This applies to **any** bindable string prop (Text's `text`, a `placeholder`, a `label`), not a single function — *"any property that can be bound to data is defined as a Dynamic\* type."*
2. **Syntax = `${…}` segments.** A template is a string interleaving literal runs with `${expr}` expressions. Inside an expression v1.0 admits **two** forms: (a) **JSON-Pointer paths** — absolute `${/user/firstName}` or relative `${firstName}` in a collection scope; (b) **function calls in an EXPRESSION grammar** identified by parentheses — `${now()}`, `${formatDate(value:${/currentDate}, format:'yyyy-MM-dd')}`, with nesting (`${…}` inside an arg). The expression-grammar function form is a **different surface** from the JSON `{call, args:{named}}` object ADR-0026 built.
3. **Escaping.** *"To include a literal `${` in a string, it must be escaped as `\${`."*
4. **Coercion (spec-exact table).** Numbers/Booleans → standard string representation; **Null/Undefined → empty string `""`**; Objects/Arrays → JSON-stringified.
5. **Reactivity.** A template *"re-resolves when an embedded path's data changes"* (the v1.0 "Text label updates as the TextField types" cross-binding example).
6. **Relative paths.** `${name}` relative in a collection scope resolves via the same scope rewrite the list uses for `{path}` and `@index`.

The repo's integration surface for this is the **literal-string arm** of `resolveValue` (`functions.ts:72`). The reactive substrate (per-path memo, `resolve`, `scopedPointer`/`itemScope`) and the conformance gate are already in place from LLD-C5/C6 and ADR-0024/0026 — this feature reuses them, it does not extend them.

## Decision

We add a **`${…}` DynamicString interpolator** as a small string-template parser layered on the existing `resolveValue` substrate, in a new **`renderer/interpolate.ts`**, hooked into the **literal-string arm** of `resolveValue` (`functions.ts`). `{path}` and `{call}` value-kind handling is byte-for-byte unchanged.

**Fork settled: (A) path-interpolation only this wave; defer the function-expression grammar.** The `${…}` expression form supports both JSON-Pointer paths and a function-call expression grammar (`name(arg:val)`). This wave ships **path interpolation** (absolute + relative-in-scope) + escaping + the full coercion table + reactivity, and **defers** the function-expression grammar (`name(arg:val)`, single-quoted string literals, nested `${…}` inside args) to its own follow-up (a new task: *function-call-in-string expression grammar*). Rationale: the function-expression form needs a real expression tokenizer (identifiers, parentheses, `arg:` named bindings, quoted literals, recursive `${…}`) that is distinct from and larger than the JSON `{call}` object ADR-0026 built, and its value is far lower than path interpolation — path interpolation is exactly what the dynamic-list demo that triggered this whole track (a people-list rendering names/roles in `Text`) needs. The parser is designed **forward-compatible** (below) so the function arm lands as a classifier extension, not a teardown.

1. **Detection — literal string with an unescaped `${`.** `isInterpolated(s)` returns true iff `s` contains a `${` not immediately preceded by a backslash. A literal string with **no** unescaped `${` falls through **byte-identical** (the `\${`-only and the ordinary cases are unchanged from today's `return value`).
2. **Scanner — one escape-aware segment split.** `interpolate` scans the template into an ordered list of **literal runs** and **`${…}` expressions**: a `\${` emits a literal `${` (the backslash is consumed); an unescaped `${` opens an expression that closes at its matching `}` (brace-depth tracked so a future nested `${…}` is parse-stable). This single scanner is the load-bearing forward-compat seam — it is unchanged when the function arm lands; only the per-expression **classifier** grows.
3. **Classifier — path now, function-expr deferred.** Each `${expr}` body is classified: a body **without** a `(` is a **JSON-Pointer path** (this wave); a body **with** a `(` is the **function-expression form** (deferred). The classifier is the single point the function arm extends.
4. **Path resolution reuses LLD-C5.** A path expression resolves through `resolve({ path: expr }, surface, itemScope)` (`binding.ts`) — the **same per-path memo** (`pathSignal`) the `{path}` value kind uses. An absolute path resolves from root; a **relative** path (no leading `/`) resolves within a list item's collection scope via `scopedPointer(expr, itemScope)` (ADR-0024) — identical rewrite, read direction, to `{path}` and `@index`. No new resolution machinery.
5. **Coercion (spec-exact).** Each resolved value is coerced to a string per fact 4: `number`/`boolean` → `String(v)`; `null`/`undefined` → `""` (the empty-string sentinel, **not** the literal `"null"`/`"undefined"`); `object`/`array` → `JSON.stringify(v)`; a resolved `string` → itself. Segments are concatenated in source order.
6. **Reactivity is free.** `resolveValue` runs **inside the existing scope-owned bound-prop effect** (`widget.ts:125`). Because each `${/path}` reads through `resolve` (the per-path computed), the bound-prop effect depends on exactly the paths the template embeds — a template with N paths re-resolves when **any** of the N changes, and an unrelated `updateDataModel` leaves it asleep (SPEC-N2 per-path waking, **reused unchanged**). This is the mechanism behind fact 5's "Text label updates as the TextField types."
7. **Conformance is a no-op — confirmed.** An interpolation template is a `typeof value === 'string'` literal, so `conformance.ts matchesType` accepts it via the ordinary string-type leg (`matchesPrimitive` → `'string'`). No `{path}`/`{call}` deferred-binding branch is involved, so **no conformance change is needed and no false `CATALOG` is raised** for a bindable string prop carrying a template. (Verified against `conformance.ts:47-49`.)

### Error / malformed handling — render literally, no error code (design decision)

The v1.0 spec defines an escaping rule (`\${`) but does **not** define a render-time error code for a malformed interpolation template, and the only render-time code the repo has is `FUNCTION` (ADR-0026), which is for an unknown/throwing function-call *value kind* — not for string parsing. We therefore **render the offending segment literally and emit no error** in two cases:

- **A deferred function-expression** (`${now()}`, body contains `(`): the unparsed `${…}` segment is emitted **verbatim** into the output string — no error. Rationale: the function arm is simply not yet implemented; emitting an error code now would burn a code (INTERPOLATION/BINDING) that the spec does not define and that the function-arm follow-up would have to un-burn. When that follow-up lands, the **same** segment resolves instead of rendering literally — a clean, additive upgrade with no behavior to retract.
- **A malformed/unterminated `${`** (no closing `}`): emitted **verbatim**, no error. The only spec escape mechanism is `\${`, so an unterminated `${` is most plausibly authoring text meant to display literally; tearing the string out of an otherwise-fine `Text` widget (or minting a new error code) is worse than showing it. This is consistent with SPEC-R4 AC2's render-time-placeholder discipline (an unresolvable binding is a placeholder, not an error) and SPEC-N4 fault isolation.

This is a defensible v1.0-faithful call, recorded as a design decision (the spec is silent on the malformed case). If a future protocol version defines an interpolation error code, that is a new ADR.

## Consequences

- **The literal-string arm of `resolveValue` is no longer a pure pass-through** — it is `isInterpolated(s) ? interpolate(s, …) : s`. Every non-`${` literal string is byte-identical to today (the guard short-circuits), so the `{path}`/`{call}` paths and all existing literal behavior are unchanged; only a string *containing* `${` changes.
- **A binding is no longer only an *object* — a template *string* is bound too, so `widget.ts isBinding` must route it into the bound-prop effect** *(discovered in build)*. Clause 6's reactivity assumes the interpolator runs inside the scope-owned bound-prop effect (`widget.ts:125`), but what *routes* a value into that effect is `isBinding`: a value it rejects is `applyProp`'d **once** (static, render-once, never reactive). A template is a plain `string`, which the prior object-only `isBinding` (`{path} | FunctionCall`) rejected — so **without** widening `isBinding` to also accept an interpolated string, a template would render once and **silently never re-resolve** (clause 6 would not hold). `isBinding` is therefore extended to `(v): v is {path} | FunctionCall | string` returning true when `v` is an object binding **or** `typeof v === 'string' && isInterpolated(v)`. Correctly scoped: a **non-template** literal string is **not** a binding and keeps the static `applyProp(once)` fast-path unchanged — only a string carrying an unescaped `${…}` is now routed reactively. (The conformance gate is still untouched — see clause 7; this widening is in the renderer's per-prop *routing* split, not the static validator.)
- **No new reactivity machinery, no new error code, no conformance change.** The interpolator is pure string-handling over the existing per-path resolver: SPEC-N2 waking, the `scopedPointer`/`itemScope` rewrite, and the bound-prop effect are all reused as-is. The Binding union (`protocol.ts`) is unchanged — a DynamicString is still a `string` literal at the type level; "is it a template?" is a **runtime** property of the string, not a new wire shape.
- **The catalog SPEC-R5 `${…}` mention is now delivered, not promissory.** ADR-0026 reconciled `formatString` away "in favor of the `${…}` follow-up"; this ADR is that follow-up's path arm. String *composition* (the `formatString` use case — `"Hello, ${/user/firstName}!"`) is now available; string *formatting* via functions in templates (`${formatDate(...)}`) waits on the deferred function-expression arm (a project catalog may still register a `{call}`-form `formatString` per ADR-0026).
- **The function-expression grammar is deferred, crisply.** A new task — *A2UI `${fn(args)}` function-call-in-string expression grammar* — owns the tokenizer (identifiers, parens, `arg:` named args, single-quoted string literals, nested `${…}` recursion) and routes a classified function-expression body to the `evaluate` arm of `functions.ts` (the same evaluator ADR-0026 built — the grammar parses *into* a `FunctionCall`, then reuses `evaluate`). The scanner (clause 2) and classifier seam (clause 3) are built to absorb it without a rewrite.
- **Malformed/deferred templates render literally** — a Text whose `text` is `${now()}` shows the literal `${now()}` until the function arm ships, and an unterminated `${` shows verbatim. This is intentional graceful degradation (no error spam, no torn UI), but it means a *typo'd path template* (`${/usr/name}` for a non-existent pointer) resolves to `""` (coercion of `undefined`), not an error — consistent with SPEC-R4 AC2 for `{path}`.
- **Stale → re-verify (on ratify + build gate):** runtime SPEC §2 (Binding — literal-string-is-conditionally-a-template note) + SPEC-R10 (`${…}` clause normative) · catalog SPEC-R5 (`${…}` delivered) · renderer LLD-C5 §5 (literal arm interpolator) + LLD-C10 §7 (`${…}` status: path arm delivered, function arm deferred) · the new `renderer/interpolate.ts` · `functions.ts` (`resolveValue` literal arm) · `conformance.ts` (confirm no change needed — string literal leg).

## Alternatives considered

- **(B) Ship the full grammar now (path + function-expression + nesting)** — rejected for this wave. The function-expression form needs a genuine expression tokenizer (parentheses, `arg:val` named args, single-quoted string literals, recursive `${…}`) that is a larger and distinct surface from both path interpolation and the JSON `{call}` object. Bundling it would delay the high-value, demo-driving path arm behind a parser whose value is far lower, and would conflate two parse problems. The scanner is built forward-compatible so (B) lands additively as a classifier extension — deferring it costs nothing structural.
- **Mint an `INTERPOLATION` / `BINDING` error code for malformed or deferred templates** — rejected. v1.0 defines no such code; the repo's only render-time code is `FUNCTION` (a value-kind concern). Rendering literally is graceful (SPEC-N4/R4 AC2), reversible (the deferred function arm upgrades the same segment with no retraction), and avoids burning a code the spec is silent on. If v1.0 later defines one, a new ADR adds it.
- **Reuse `FUNCTION` for a `${fn()}` segment encountered before the function arm exists** — rejected. `FUNCTION` is for an unknown/throwing `{call}` value kind that the renderer *attempted to evaluate*; a `${fn()}` template the interpolator does **not yet parse** has not been evaluated and is not unknown — it is unimplemented. Emitting `FUNCTION` would mislead the server into thinking its catalog function is missing. Literal render correctly signals "not interpreted."
- **Coerce `null`/`undefined` to `"null"`/`"undefined"`** (the naïve `String(v)`) — rejected: violates the spec coercion table (fact 4), which mandates the empty string for both. A template like `"${/middleName} "` over absent data must render `" "`, not `"undefined "`.
- **Detect a template by parsing the schema `DynamicString` union (a new wire shape)** — rejected. v1.0 says interpolation is triggered by the string *content* at runtime (`${…}` present), not by a distinct serialized type; a DynamicString literal is still a plain JSON string on the wire. Adding a wire shape would diverge from the spec and from the `protocol.ts` Binding union (which stays a three-arm `literal | {path} | {call}`).
- **Add a per-template memo (cache the parsed segment list per string)** — rejected as premature. The scan is O(len) over a short prop string run once per `applyProp` inside an already-memoized effect that only re-runs when an embedded path changes; the per-*path* memo already prevents redundant pointer walks. A parse cache adds a map + key discipline for no measured cost. Revisit only if profiling shows the scan hot.
