## Context

Task #14's title conflated two unrelated features; the decomposition split them (team-lead-confirmed): **#14a — the `checks` feature** (this ADR) and **#14b — error-vocab reconciliation** (`VALIDATION_FAILED` for *schema* errors + `FUNCTION`→`INVALID_FUNCTION_CALL`, task #19, no overlap with this). Grounded against the repo at HEAD and the **A2UI v1.0** validation-section facts fetched from a2ui.org and supplied by team-lead (Constraint **C1**):

1. **`checks` is a component-level array of `{call, args, message}`.** On input components (TextField, Checkbox, Slider, ChoicePicker, DateTimeInput) **and** Buttons: `"checks": [{ "call":"required", "args":{ "value":{"path":"/formData/zip"} }, "message":"Zip code is required" }]`. Each entry is a function-call (the ADR-0026 `{call,args}` shape) plus a human `message`. **Shape ambiguity:** the field example is **flat** `{call,args,message}`; the Button example wraps the call as `{condition:{call,args}, message}`. (Reconciled below.)
2. **Semantics: a check's call returns TRUE = valid, FALSE = invalid.** A failed check's `message` *"can be displayed when the component is rendered."* **Timing is spec-silent** (input/blur/submit/continuous undefined) → a design decision.
3. **Surfacing is client-side INLINE DISPLAY** — the failed message shows **on the component**, not via any error channel.
4. **Checks emit NO server error.** *"Servers never receive notifications of failed checks unless explicitly included in an action's context."* `VALIDATION_FAILED` is **exclusively** the *schema*-validation code (malformed protocol JSON — "Expected stringOrPath, got integer"), **not** form validation. So this feature touches **no** `ErrorCode` and **no** protocol error path. (ADR-0026's fact-7 note anticipated a `VALIDATION_FAILED`-on-check link; the v1.0 spec governs — checks ≠ `VALIDATION_FAILED`.)
5. **Button auto-disable:** *"Buttons can also define checks. If any check fails, the button is automatically disabled."* A client-side gate preventing submission.

The repo already has every mechanism this rides on — the discovery that **right-sizes** the feature:
- **`evaluate` (LLD-C10, ADR-0026)** runs a `{call,args}` against the bound catalog, recursively resolving a `{path}` arg through the per-path memo. A check is exactly a `{call,args}` — so checks **reuse** it.
- **ui-text-field already has the field-error display channel** — a control-managed `aria-describedby` message node + a `:state(user-invalid)` danger-border ladder driven by `formValidity().message` (text-field.ts, ADR-0014 cl.4). The node is currently `hidden` (aria-only; "the visible wrapper is ui-field at G7"). So #14a is **not** "build a field-error channel" — it is a *custom-validity seam* + the renderer wiring + making the existing node visible.
- **The host's `createWidget` wrapper already strips + wires a non-prop component-level key (`action`) into a listener** (`renderer.ts` `#makeHostCreateWidget`/`#wireAction`), with `el`/`node`/`surface`/`scope`/`itemScope`/`ac` in scope. `checks` is the **same kind** of key, wired the **same way**.
- **`ui-button` + `ui-text-field` both ship a reflecting `disabled` prop** — button auto-disable is `el.disabled = true`, no new control work.

## Decision

We deliver the `checks` feature as a renderer-driven, reactive, **client-side-only** validation layer in a new **`renderer/checks.ts`** controller, plus one small components-layer seam (`setCustomValidity` on `UIFormElement`) and the user-ratified visible-message change to ui-text-field. **No protocol/`ErrorCode` change; no server error emission.**

1. **A `Check` internal type, two accepted wire shapes (Fork 1 — Postel).** `Check = { call: string; args?: …; message: string }`. The **flat** form `{call,args,message}` is canonical (it matches `evaluate` directly); the Button **`condition`-wrapper** `{condition:{call,args}, message}` is **tolerantly read** — a reader unwraps `condition`→`{call,args}` — symmetric with ADR-0011's action-shape Postel reader. One internal type, both wire shapes accepted. The unwrap keys on the **`condition`** wrapper key **only** — it is orthogonal to the check's **call name**: a combinator check `{call:'and', args:{…}, message}` is a flat check whose call is `and`, NOT a `condition` wrapper, so it flows through `evaluate` as an ordinary catalog function (and `and`/`or`/`not` correctly `FUNCTION`-error against the validators-only default catalog until a catalog ships them — conformant, like `now`/`formatDate`).
2. **`checks` is a component-level RESERVED key, not a bindable prop.** Like `action`, `checks` is read off the node by the host wrapper, **never** `applyProp`'d. It joins `RESERVED` in `widget.ts` **and** `conformance.ts` (so a `checks` array on a node raises **no** false `CATALOG` unknown-property). **`RESERVED` is the realized recognition — a per-component catalog *declaration* of which components accept `checks` is DEFERRED** (discovered in build, see Consequences): the catalog loader's `validatePropDef` requires `mapsTo: string` on every declared property, so a no-`mapsTo` `checks` marker is infeasible without a validator extension. A `checks` on a component that is neither an input nor a Button is therefore accepted structurally (RESERVED) and the controller **no-ops** on it (no `setCustomValidity`, no `disabled`) — harmless for a v1.0 *renderer*, which must not reject otherwise-valid payloads.
3. **A generic `checks` controller (`checks.ts`), wired in the host `createWidget` wrapper.** After `base(node,…)` builds the control, `#wireChecks(el, node, surface, scope, ac)` (sibling to `#wireAction`) reads `node.checks`. For each check it runs `evaluate(check.{call,args}, surface, itemScope, emitError, registry)` (the ADR-0026 evaluator) inside a **`scope`-owned effect**, interpreting **TRUE = valid, FALSE = invalid** (Fork: a check call returning a `{valid}` result — the `required`/`email`/`regex` shape — is read via `.valid`; a bare boolean is read directly). Generic — no component name appears; opt-in by `node.checks` presence.
4. **Reactive timing (Fork 2 — reactive; spec-silent so a design decision).** The checks effect runs inside `scope` and reads each `{path}` arg through `evaluate`→`resolveValue`→`resolve` (the per-path memo), so a check **re-evaluates when its bound data changes** (SPEC-N2, reused) — live validation. The *visible* danger treatment still rides ui-text-field's existing **`:state(user-invalid)` timing** (the danger border/message appear only after the first blur/change, ADR-0014 cl.2c) — so the value is validated reactively but the user is not scolded before first interaction. The effect/listener is owned by the item-or-surface `scope`/`AbortController` (SPEC-N3, leak-free; a list item's checks die with the item).
5. **Input target — `setCustomValidity` seam on `UIFormElement` (components layer).** A new signal-backed `setCustomValidity(message: string)` (native-parity name) sets a `#customValidity` signal; the base's existing validity effect **merges** it with the subclass `formValidity()` verdict: if `formValidity()` is already invalid, native wins (e.g. `valueMissing`); else a non-empty custom message produces `{valid:false, flags:{customError:true}, message}` (Fork B = B1, add-alongside — native + custom coexist, the present message taking the display slot). The renderer drives it without the control knowing about A2UI — `el.setCustomValidity(firstFailingCheck.message)` on fail, `el.setCustomValidity('')` when all pass. ui-text-field's existing message node + `user-invalid` border then display it for free.
6. **Visible message — the A1 change to ui-text-field (extends ADR-0014, user-ratified).** The control-managed message node (`.ui-text-field-message`, today `hidden`) becomes **visible when non-empty**: the user-invalid effect that sets `message.textContent` also toggles `message.hidden` (visible iff the text is non-empty), and `text-field.css` gives it a visible danger-text treatment (`--c-danger` ink, small type, gated by `:state(user-invalid)` like the border). When valid, the node returns to `hidden`/empty. The `aria-describedby` wiring is unchanged; this only makes the already-present node visible (ADR-0014's contenteditable/interaction-state Decision stands).
7. **Button target — auto-disable (`disabled` reflecting prop, exists).** For a Button with `checks`, the controller drives `el.disabled = true` when **any** check fails, and restores the node's declared/bound `disabled` when all pass. No new control surface — `ui-button`'s reflecting `disabled` (+ its `tabbable`/`pressActivation` disabled gating, ADR-0010) already does the rest.
8. **Fault isolation (reused).** A check naming an unknown/throwing function rides `evaluate`'s existing `FUNCTION` emit + `undefined` (treated as invalid for the gate); a sibling check, prop, or widget is unaffected (SPEC-N4). This is the **only** error path, and it is the *function-resolution* error (ADR-0026), not a check-failure error — a *failing* check emits nothing.

## Consequences

- **Pure client-side UX — no error wiring, no protocol change.** The checks controller drives a control's validity/disabled and nothing else; it emits no server error and adds no `ErrorCode`. `protocol.ts` is untouched. (The `VALIDATION_FAILED`/`INVALID_FUNCTION_CALL` vocab is task #19, fully independent.)
- **The mechanism is a thin reuse layer.** `checks.ts` is `evaluate` (existing) + `setCustomValidity` (new, ~10 lines on the form base) + `el.disabled` (existing). The reactivity (SPEC-N2), the leak-free teardown (SPEC-N3), the function dispatch + fault isolation (ADR-0026), and the field-error display (ADR-0014) are all reused. The only genuinely new surfaces are the form-base custom-validity seam and the visible-message CSS.
- **`checks` joins `RESERVED` at two gates** (`widget.ts`, `conformance.ts`) — a node may now legally carry a `checks` array without a false `CATALOG`.
- **Per-component catalog *declaration* of `checks` is DEFERRED (discovered in build).** The plan was for the catalog to *declare* which components accept `checks` (SPEC-R4). The catalog loader's `validatePropDef` requires `mapsTo: string` on **every** declared property, so a no-`mapsTo` `checks` marker throws `CatalogError: property "Button.checks".mapsTo must be a string`. The realized recognition is therefore **`RESERVED`-only** (no per-component declaration this wave). Consequence: a `checks` on a component that is neither input nor Button is accepted structurally and the controller **no-ops** (the target has neither `setCustomValidity` nor `disabled`) — acceptable for a v1.0 renderer (it does not reject valid payloads; it gracefully no-ops a misplaced one). Tightening this to a declared per-component allow-set needs a **catalog-schema follow-up** (a `mapsTo`-optional component-level construct in `validatePropDef`) — **not booked**; flagged here as the owning constraint.
- **ui-text-field's message node is now visible** (extends ADR-0014) — a components-layer control + CSS change with a **browser-smoke DoD** (the visible danger-text treatment + the `:state(user-invalid)` gating render in a real engine). This slightly precedes the `ui-field` G7 wrapper, by user ratification (visible-now over deferral); `ui-field`, when it lands, can host/relayout this message rather than re-introduce it.
- **Native `required` and A2UI `checks` coexist (B1).** A TextField may carry both the boolean `required` (native `valueMissing`) and a `checks` array; the base merges — native invalid wins, else custom. B2 (migrate required→a checks-only expression) is a separate v1.0-fidelity follow-up (booked), out of scope here.
- **Button auto-disable is a client gate, not a validity verdict** — a Button is not a `UIFormElement`, so it has no `setCustomValidity`; its checks drive `disabled` directly. The asymmetry (input → message, button → disable) is the v1.0 model (facts 3 + 5), not an inconsistency.
- **`{path}`-bound `disabled` + `checks` on the same Button is a deferred tension (discovered in build).** The controller restores `Boolean(node.disabled)` when all checks pass — correct for a static boolean `disabled`. But if a Button **also** binds `disabled` to a `{path}`, two effects (the checks effect and the bind-prop effect) both write `el.disabled` (a write race), and `Boolean({path-object})` is always truthy (the raw binding object, not its resolved value). The v1.0 model is that **checks OWN `disabled`** when present, so this is a benign edge for the common case (a Button uses *either* a bound `disabled` *or* `checks`, rarely both). Combining a bound `disabled` with `checks` on one Button is a **deferred follow-up tension** — recorded here so it is not a silent latent bug; not fixed this wave.
- **Stale → re-verify (on ratify + build gate):** catalog SPEC-R4/R5 (`checks` construct — RESERVED recognition, declaration deferred) · renderer LLD-C10 §7 / LLD-C13 (the checks controller) · goals §G4 (the form-base custom-validity seam) · the new `checks.ts` · `form.ts` (`setCustomValidity` + merge) · `text-field.{ts,css}` + its `.md`/descriptor (visible message) · `conformance.ts` + `widget.ts` (`RESERVED`).

## Alternatives considered

- **Emit `error{code:VALIDATION_FAILED}` on a failed check** (my earlier draft, and ADR-0026 fact-7's anticipation) — **rejected**, corrected by the v1.0 facts. Checks are purely client-side; *"servers never receive notifications of failed checks."* `VALIDATION_FAILED` is the *schema* code for malformed protocol JSON, an unrelated dev-time concern (task #19). Emitting it on a check failure would conflate a UI feature with a protocol-conformance code and send the server noise the spec says it must not get.
- **Accept only the flat `{call,args,message}` shape (reject the `condition` wrapper)** — rejected. The v1.0 Button example uses the `condition` wrapper; rejecting it would mis-validate a spec-conformant payload. The tolerant-read (unwrap `condition`) costs one branch and matches the repo's established Postel discipline (ADR-0011 action shape). Canonical-flat keeps the internal type single.
- **Build a new a2ui-layer error-display element instead of reusing ui-text-field's validity surface** — rejected. The control already owns its invalid state + message node + `user-invalid` timing (ADR-0014); a parallel a2ui display would duplicate it, fight the control's own `:state(user-invalid)`, and break the components/a2ui layer boundary. Driving the control's *native* validity (via `setCustomValidity`) keeps one display path and one timing source.
- **Defer the visible message to `ui-field` (A2)** — rejected by user ratification (A1). `ui-field` is unscheduled; gating #14a's visible display on it would ship a feature whose message only screen-readers see. Making the existing node visible is small and reuses the node; `ui-field` later relayouts it.
- **Run checks only at action-commit (gate on submit, not reactive)** — rejected. The spec is silent on timing, but a reactive evaluation (free via the per-path memo) gives live validation, and the *display* is still gated by `user-invalid` timing (no pre-interaction scolding), so reactive costs nothing and is the better UX. Action-commit-only would make a check feel stale (the message lingers after the user fixes the value).
- **Put `setCustomValidity` on each control instead of the form base** — rejected. Validity is a `UIFormElement` concern (it owns `formValidity()`→`internals.setValidity`); a per-control seam would duplicate the merge logic on every input control. One base seam, inherited, keeps the renderer's driver control-agnostic.
