## Context

Task #19 (#14b, split from #14): align our error vocabulary to A2UI v1.0. Grounded against the repo at HEAD and the **A2UI v1.0** ErrorCode facts fetched from a2ui.org and supplied by team-lead (Constraint **C1**):

1. **v1.0 defines EXACTLY TWO client→server error codes — and the protocol governs ONLY the wire.** The `code` enum is the closed set `{ INVALID_FUNCTION_CALL, VALIDATION_FAILED }`, and crucially *"the spec defines only WIRE-level error messages; INTERNAL validation is NOT part of the protocol specification."* So our 8 internal codes are **legitimately ours** — the spec constrains only the client→server `error` envelope, not our diagnostic taxonomy. (This is the spec license for fork B.)
   - **`INVALID_FUNCTION_CALL`** — *"if the requested function is configured `clientOnly`, OR if the function is not registered at all, the client MUST immediately reject the call and return `code: INVALID_FUNCTION_CALL`."* Covers the clientOnly-boundary **and** the unknown/unregistered-function case. (A *throwing* function mid-eval is a spec **gap** — undefined; emitting `INVALID_FUNCTION_CALL` is our defensible choice.)
   - **`VALIDATION_FAILED`** — the **only** validation code, so it is the catch-all for ALL structural/protocol validation (*"Expected stringOrPath, got integer"* is one example; there is no finer wire code) — **not** form validation.
2. **The wire error shape** = exactly `{ code (req), message (req), surfaceId ⊕ functionCallId }` — **no `path`**. The contextID is **TIED to the code** (not a free XOR): `INVALID_FUNCTION_CALL` **requires `functionCallId`** (excludes `surfaceId`); `VALIDATION_FAILED` **requires `surfaceId`** (excludes `functionCallId`). `message` is free-form.

Our repo today has a **richer** internal vocabulary — **8 codes** (`protocol.ts ErrorCode`): `PARSE · SCHEMA · CATALOG · CATALOG_UNKNOWN · IDGRAPH · POINTER · VERSION_UNSUPPORTED · FUNCTION`; `A2uiError = { code, surfaceId?, path?, message }` (no `functionCallId`). These codes are **dual-use**, which is the load-bearing distinction:
- **Internal consumers need the fine codes.** `validate.ts` (the validator, LLD-C11) emits fine `Failure` codes that drive (a) the renderer's render/skip logic and (b) **corpus admission's verdict** — and `validate.ts` is the **shared** validator (SPEC-N6 parity; corpus re-exports it). A corpus record's admission distinguishes a `SCHEMA` failure from an `IDGRAPH` failure; collapsing the codes would gut that diagnostic.
- **Only the WIRE emit is constrained to the v1.0 two codes.** Every client→server error flows through **one chokepoint** — `renderer.ts #emit` (the outbound `A2uiClientMessage` sink; PARSE/CATALOG_UNKNOWN/IDGRAPH/the render-time `emitError` all route here).

So the codes serve two roles — a rich internal taxonomy **and** the wire contract — and the two roles have different requirements.

## Decision

**Fork B — keep the rich internal taxonomy; map to the v1.0 two-code contract at the single wire boundary.** Our 8 codes stay as the **internal** `ErrorCode` (the validator, the corpus verdict, the render-time `emitError`, and their tests all keep full precision, unchanged). The outbound `A2uiErrorMessage` carries a **`WireErrorCode`** (the v1.0 two codes), produced by a `toWireError` map applied at `renderer.ts #emit`.

1. **Two code types + a discriminated wire shape.** `protocol.ts` keeps `ErrorCode` (the 8 internal codes) and adds `WireErrorCode = 'INVALID_FUNCTION_CALL' | 'VALIDATION_FAILED'`. `A2uiError` (internal) is unchanged (`code: ErrorCode`, `surfaceId?`, `path?`, `message`). The outbound `A2uiWireError` is a **discriminated union** carrying exactly the v1.0 wire fields (**no `path`**): `{ code:'VALIDATION_FAILED'; surfaceId: string; message: string } | { code:'INVALID_FUNCTION_CALL'; functionCallId: string; message: string }`. `A2uiErrorMessage.error` becomes `A2uiWireError`.
2. **The map — ALL wire errors → `VALIDATION_FAILED` + `surfaceId` this wave (the flow-grounded resolution).** `toWireError(internal: A2uiError): A2uiWireError`:

   | internal code | → wire code | rationale |
   |---|---|---|
   | `PARSE` · `SCHEMA` · `CATALOG` · `CATALOG_UNKNOWN` · `IDGRAPH` · `POINTER` · `VERSION_UNSUPPORTED` · **`FUNCTION`** | `VALIDATION_FAILED` (+ `surfaceId`) | every client→server error we emit is a **message-validation** failure. **`FUNCTION` included** (the flow finding, below): our `FUNCTION` emits are **render-time binding-evaluation** failures (`@index` misuse, an unknown/throwing catalog function *referenced in a binding*) — the *message references an invalid function*, exactly parallel to `CATALOG` (the message references an unknown component). They are **not** the spec's `INVALID_FUNCTION_CALL`, which is a **server-initiated** function-call failure (`clientOnly`/unregistered on an invocation) — a path the repo **does not have**. |

   **Flow finding (grounded):** `FUNCTION` errors reach the wire today (`emitError → #emit`, renderer.ts:279), but the repo has **no server-initiated function-invocation path** — the inbound message set is `createSurface`/`updateComponents`/`updateDataModel`/`deleteSurface`/`actionResponse` (no "invoke function"), and `clientOnly` appears nowhere. So every `FUNCTION` emit is a render-time message-validation failure → `VALIDATION_FAILED`. The `message` carries the function detail (with the `path` locus folded in, clause 4).
3. **The contextID is TIED to the code — every emit this wave is `VALIDATION_FAILED` + `surfaceId` (unambiguously conformant).** v1.0 binds the contextID to the code: `VALIDATION_FAILED` **requires `surfaceId`** (excludes `functionCallId`); `INVALID_FUNCTION_CALL` **requires `functionCallId`** (excludes `surfaceId`). The wire type is a **discriminated union** — `{ code:'VALIDATION_FAILED'; surfaceId; message } | { code:'INVALID_FUNCTION_CALL'; functionCallId; message }` — both arms modeled (forward-ready). But **only the `VALIDATION_FAILED` arm is EMITTED this wave**: we always have `surfaceId`, and the `INVALID_FUNCTION_CALL` arm cannot be emitted conformantly without a `functionCallId`, which requires a server-initiated function-call path (absent) AND a modeled call id (absent). So **`INVALID_FUNCTION_CALL` is RESERVED, not emitted** — deferred to **#23** (the server-initiated function-call path + `functionCallId` modeling, bundled). This corrects the earlier "`FUNCTION → INVALID_FUNCTION_CALL` + `node.id` as `functionCallId`" lean — `node.id` is not a function-call id and our errors are not server-initiated; mapping them to `VALIDATION_FAILED` is both conformant and semantically right.
4. **`path` is NOT a wire field — fold it into `message`.** v1.0's wire shape is exactly `{ code, message, surfaceId ⊕ functionCallId }` — **no `path`**. The internal `A2uiError` keeps `path` (the validator/render locus); `toWireError` **folds** a present `path` into `message` (e.g. `"unknown component type X (at root/field)"`) so the locus survives for the server, then drops the field. The wire type has no `path`.
5. **The shared validator + corpus parity are UNTOUCHED.** `validate.ts` keeps its fine `Failure` codes; the corpus admission verdict is byte-for-byte unchanged (SPEC-N6 parity holds). The map lives **only** at the renderer's wire boundary — the validator never sees it.

## Consequences

- **Maximally conformant wire, full internal precision — and the smallest test ripple.** The server receives **exactly** the two v1.0 codes; the renderer + validator + corpus keep their 8-code precision. Because the internal codes are unchanged, the **~33 internal `emitError` assertions** (functions.test 19, interpolate 7, checks 6, action 1 — all asserting the internal `FUNCTION`) **do NOT change**; only the **~5 wire-emit assertions** in `renderer.test.ts` (which assert the outbound message `code`) flip to the mapped code (`CATALOG`/`PARSE`/`CATALOG_UNKNOWN`/`IDGRAPH` → `VALIDATION_FAILED` + `surfaceId`, the `path` folded into `message`). (Option A — collapsing the union — would have rippled all ~33 internal assertions, gutted the validator's fine codes, and broken corpus parity. The map localizes the change.)
- **`FUNCTION` is NOT renamed** — it stays the internal render-time code (ADR-0026 stands); the wire translation is the new layer. The checks fault-gate (ADR-0029 §8, "`undefined` = FUNCTION error → treated invalid") and every `evaluate` emit are unchanged.
- **The server receives `VALIDATION_FAILED` for EVERY error this wave** (including function-eval failures), with the fine internal code's detail in `message` (the `path` locus folded in). The code-level granularity (CATALOG vs IDGRAPH vs the function-eval case) lives in `message`; this is the v1.0 contract (two codes + a free-form message), and `surfaceId` is always its required, available context — so every emit is unambiguously conformant.
- **`INVALID_FUNCTION_CALL` is RESERVED, not emitted this wave** — it requires a `functionCallId` (the code↔contextID tie excludes `surfaceId`), which in turn requires (a) a **server-initiated function-call path** (the repo has none) and (b) a **modeled call id** (absent). Both are deferred to **#23**. The wire type models the `INVALID_FUNCTION_CALL` arm (forward-ready) but no runtime path produces it. This corrects two earlier leans (both pre-full-grounding): "emit `surfaceId` for `INVALID_FUNCTION_CALL`" (violates the tie) and "use `node.id` as the `functionCallId`" (`node.id` is not a call id; our errors are not server-initiated). The flow finding settles it: render-time binding-eval function errors are message-validation failures → `VALIDATION_FAILED`.
- **`path` leaves the wire** — v1.0's wire shape is `{code, message, surfaceId ⊕ functionCallId}` only; `toWireError` folds a present `path` into `message` and drops the field. Internal `A2uiError` keeps `path` (validator/render locus). The renderer.test.ts wire assertions drop their `path:` field-match and look for the locus inside `message`.
- **`VERSION_UNSUPPORTED`/`CATALOG_UNKNOWN` map to `VALIDATION_FAILED`** — arguably "capability/config" rather than "validation," but the two-code enum offers no third bucket and they are not function-call errors, so `VALIDATION_FAILED` is the only conformant wire code. Recorded as a forced mapping.
- **No `node.id` threading needed this wave** — because every emit is `VALIDATION_FAILED` + `surfaceId` (which the FUNCTION emit sites already carry), the build does **not** thread `node.id` into the `FUNCTION` error. The map is a pure code+context transform at `#emit`. (The node/call-id threading lands with #23's `INVALID_FUNCTION_CALL` path.)
- **Stale → re-verify (on ratify + build gate):** runtime SPEC §5.2 (wire shape = `{code, message, surfaceId ⊕ functionCallId}`, no path; the 8 = internal taxonomy + the map table; only `VALIDATION_FAILED` emitted) + SPEC-R10/R11 · renderer LLD §9 (error table + the wire-map column) + LLD-C13 (`#emit`) · protocol.ts (`WireErrorCode`/discriminated `A2uiWireError`/`toWireError`) · renderer.ts (`#emit` map) · renderer.test.ts (the ~5 wire assertions) · ADR-0026 back-link.

## Alternatives considered

- **(A) Collapse the internal union to the two wire codes** (every emit site emits only `INVALID_FUNCTION_CALL`/`VALIDATION_FAILED`) — rejected. It coarsens the **internal** taxonomy that the **shared validator** and **corpus admission verdict** depend on (a corpus record could no longer distinguish a `SCHEMA` from an `IDGRAPH` failure — SPEC-N6 parity gutted), and it ripples **all ~33 internal `FUNCTION`/code assertions** plus the validator's `Failure`-code tests. Maximal conformance on the wire at the cost of internal precision and a large, low-value test churn. B achieves the same wire conformance with a thin boundary map and a ~5-assertion ripple.
- **Make the wire error carry a richer code or extra fields (a vendor extension)** — rejected. The spec is explicit: `{code, message, surfaceId ⊕ functionCallId}`, two codes, **no `path`**. Specificity belongs in the free-form `message` (the `path` locus folded in), not a non-standard code/field a v1.0 server would not understand.
- **Rename internal `FUNCTION` → `INVALID_FUNCTION_CALL`** (a 1:1 internal rename) — rejected. It conflates the internal taxonomy with the wire contract (the internal code would then carry a wire name), ripples every internal `FUNCTION` assertion for no internal benefit, and still leaves the other 7 codes needing a wire bucket. Keeping `FUNCTION` internal + mapping at the boundary is cleaner and smaller.
- **Emit `surfaceId` (not `functionCallId`) for `INVALID_FUNCTION_CALL`, deferring functionCallId modeling** — rejected (this was my pre-full-grounding lean). v1.0 **ties** the contextID to the code: `INVALID_FUNCTION_CALL` **excludes** `surfaceId` and **requires** `functionCallId`. So a `surfaceId`-carrying function error is non-conformant; `functionCallId` (= `node.id` this wave) is required, not deferrable. Modeling a *finer* per-call id stays a refinement (#23).
- **Keep `path` on the wire as an optional field** — rejected (also a pre-full-grounding lean). v1.0's wire shape has no `path`; carrying it would emit a field a conformant server does not expect. Folding it into `message` preserves the locus without a non-standard field.
- **Treat the codes as "internal-only" and stop emitting them to the server** — rejected. The server legitimately needs to know a message failed (PARSE/SCHEMA/CATALOG are real client→server signals); the fix is to emit the **conformant** wire code, not to suppress the error.
